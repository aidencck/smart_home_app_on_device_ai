# 智能家居端侧模型：数据评估与合成规则逆向推导 (Data Engineering Logic)

在端侧大模型微调中，**“Garbage In, Garbage Out”** 是铁律。要确保我们合成的 SFT (Supervised Fine-Tuning) 数据能够让 0.5B 这样的小模型精准控制智能家居，我们需要建立一套严密的**评估体系**，并从业务目标**逆向推导**出数据合成的黄金规则。

---

## 一、 如何评估合成数据“能否满足项目需求”？

我们对端侧模型的核心需求是：**在极低算力下，将自然语言100%转化为确定性的设备控制 JSON，且绝不越权。**
因此，数据的评估维度必须围绕以下四点：

### 1. JSON 格式的刚性遵循度 (Format Strictness)
*   **评估指标**：验证集中，模型输出能否被 `json.loads()` 成功解析的比例。
*   **需求映射**：Flutter 端依赖反序列化后的 `AgentIntent` 对象执行设备调用。如果模型输出带有 Markdown 标记（如 ````json`）或多余的解释性废话（如“好的，已经为您打开”），将导致解析失败。

### 2. 意图映射的准确率 (Intent Accuracy)
*   **评估指标**：针对验证集，计算 `device_id`、`action` 和 `value` 的精确匹配率 (Exact Match)。
*   **需求映射**：当用户说“太刺眼了”，模型必须精准映射到 `{"device_id": "light_1", "action": "turn_off"}`，而不是去调整空调。

### 3. 安全与越界拒绝率 (Safety & Out-of-Domain Rejection)
*   **评估指标**：当输入与智能家居无关的指令（如“讲个笑话”、“如何制造炸弹”）时，模型输出 `{"action": "none"}` 的成功率。
*   **需求映射**：端侧算力宝贵，绝不能用来做闲聊。模型必须具备强烈的“职责边界感”。

### 4. 复杂状态下的条件推理 (Conditional Reasoning)
*   **评估指标**：在 System Prompt 提供的设备状态列表中存在多个同类设备（如两个灯）时，模型能否根据上下文或常识正确选择。
*   **需求映射**：体现智能体不仅仅是声控开关，而是有逻辑判断能力。

---

## 二、 逆向推导：数据合成的 5 条黄金规则

基于上述严苛的评估标准，我们反推到 `data_synthesis.py` 脚本时，必须遵循以下数据合成规则：

### 规则 1：绝对剥离“废话” (No Chatty Responses)
*   **推导逻辑**：为了保证 100% JSON 解析率，并最大化端侧推理速度（Tokens/s），模型不需要礼貌，只需要干活。
*   **合成要求**：`assistant` 的 `content` **必须仅包含纯 JSON 字符串**。
*   ❌ **错误合成**：`{"role": "assistant", "content": "好的主人，已经为您生成指令：\n```json\n{\"action\": \"turn_on\"}\n```"}`
*   ✅ **正确合成**：`{"role": "assistant", "content": "{\"device_id\": \"ac_1\", \"action\": \"turn_on\"}"}`

### 规则 2：System Prompt 必须包含“动态设备快照” (Dynamic Context)
*   **推导逻辑**：模型不能死记硬背 `device_id`，它必须学会从当前的上下文中“寻找”可用设备。
*   **合成要求**：每一条合成数据的 `system` 角色中，必须随机注入当前可用的设备列表。
*   ✅ **正确合成**：`{"role": "system", "content": "你是一个智能家居端侧Agent。当前设备列表: [{'id': 'light_1', 'name': '主卧灯'}, {'id': 'ac_2', 'name': '次卧空调'}]"}`

### 规则 3：故意注入“模糊与常识”指令 (Fuzzy Intent Injection)
*   **推导逻辑**：如果只合成“打开灯”、“关空调”的数据，不需要大模型，正则表达式就能做。必须让模型学习“意图推断”。
*   **合成要求**：按比例（如 30%直接指令，50%模糊指令，20%闲聊拒答）生成数据。
*   ✅ **正确合成示例**：
    *   *输入*：“我准备看电影了。”
    *   *输出*：`{"device_id": "light_1", "action": "turn_off"}` (隐式推理：看电影需要暗环境)

### 规则 4：构建负样本与边界测试 (Negative Sampling)
*   **推导逻辑**：为了达到高安全与越界拒绝率，训练集中必须有大量诱导模型犯错的“负样本”。
*   **合成要求**：加入各类与家居无关的刁钻问题，并将其目标输出强制对齐为无动作。
*   ✅ **正确合成示例**：
    *   *输入*：“帮我写一段 Python 快速排序代码。”
    *   *输出*：`{"action": "none"}`

### 规则 5：数据分布的“长尾覆盖” (Long-Tail Coverage)
*   **推导逻辑**：用户说话带有口音、方言、错别字或倒装句。
*   **合成要求**：在生成 Prompt 脚本（交由 GPT-4/Claude 生成数据时）中，明确要求使用各种句式。
*   ✅ **正确合成示例**：
    *   *正常*：“把温度调到26度。”
    *   *倒装*：“26度，空调调一下。”
    *   *口语*：“屋里有点冻脚啊，开点暖气呗。”

---

## 三、 总结：从合成到评估的闭环

通过以上逆向推导，我们的工作流应当是：
1. **定义评估集**：先人工手写 200 条极其刁钻的测试用例（包含上述各类情况）。
2. **制定合成规则**：将上述 5 条规则写入大模型（GPT-4）的 Prompt 中，让其批量生成 10k+ 条训练数据。
3. **微调与评估**：使用这些数据微调 Qwen-0.5B，然后在第 1 步的评估集上测试 JSON 成功率、拒答率等指标。
4. **迭代**：如果某类问题（如倒装句）准确率低，则回到第 2 步，在合成脚本中增加倒装句的生成权重，再次合成并训练。
