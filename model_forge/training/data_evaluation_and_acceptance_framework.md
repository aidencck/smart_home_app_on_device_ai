# 智能家居端侧模型：数据评估与验收体系指南 (Data Evaluation & Acceptance Framework)

本文档旨在为团队提供一套**具体、可量化、可执行**的端侧模型微调数据评估与验收体系。它是连接“数据工程”与“模型部署”的桥梁，用于指导团队如何在日常任务中评估数据集质量，并作为最终模型能否上线端侧的验收标准。

---

## 🎯 一、 评估体系的四大核心指标 (Model Acceptance KPIs)

> **注**：本部分为端侧 AI 全生命周期指标体系（参见 [`full_lifecycle_ai_architecture_solution.md`](../../docs/architecture/full_lifecycle_ai_architecture_solution.md) 第 6 节）中**阶段二（模型能力验收）**的详细执行标准。

我们对端侧模型（如 Qwen2.5-0.5B 或 Gemma-2B）的验收，不再依赖传统的 BLEU 或 ROUGE 等文本相似度指标，而是直接与**端侧业务成功率**挂钩。

### 1. JSON 格式刚性解析率 (FSR - Format Strictness Rate)
*   **定义**：模型输出的内容，能够直接被 Python `json.loads()` 或 Dart `jsonDecode()` 无错解析的比例。
*   **验收标准**：**≥ 99.5%**
*   **测试方法**：遍历验证集的模型输出，统计抛出 `JSONDecodeError` 的次数。任何包含 ```json 前缀、额外标点、或解释性废话（如“好的”）的输出均视为失败。

### 2. 意图精确匹配率 (IEM - Intent Exact Match)
*   **定义**：解析成功的 JSON 中，`device_id`、`action` 和 `value` 三个核心字段与 Ground Truth 完全一致的比例。
*   **验收标准**：**≥ 95.0%**
*   **测试方法**：
    *   直接指令测试（如“关灯”）：要求 100% 匹配。
    *   模糊指令测试（如“我有点冷”）：要求匹配到 `ac_x` 且 `action` 为调高温度。

### 3. 安全与越界拦截率 (OOD-R - Out-of-Domain Rejection Rate)
*   **定义**：当用户输入与智能家居控制无关（如写代码、问百科、违规内容）时，模型输出 `{"action": "none"}` 的比例。
*   **验收标准**：**≥ 98.0%**
*   **测试方法**：使用专门的“负样本/越界”验证子集进行测试，若模型尝试回答越界问题或产生幻觉操作设备，即视为失败。

### 4. 动态上下文抗干扰率 (DCR - Dynamic Context Resilience)
*   **定义**：当 System Prompt 中的设备列表发生变化（如设备离线、新增同名设备）时，模型不产生幻觉（不调用不存在的设备 ID）的能力。
*   **验收标准**：**≥ 99.0%**
*   **测试方法**：在测试集中故意构造“指令中提及了设备，但 System 设备列表中未提供该设备”的陷阱题，模型必须输出 `{"action": "none"}` 或报错意图。

## 🛠️ 二、 团队任务推进与验证流 (Task Workflow)

为了将上述指标落地，数据工程师与算法工程师需遵循以下迭代流：

### Step 1: 构建 Benchmark 黄金测试集 (Golden Dataset)
*   **行动**：人工手写而非机器生成，构建一个包含 500 条数据的纯净测试集 `benchmark_v1.jsonl`。
*   **数据分布**：
    *   直接控制指令：40% (200条)
    *   模糊与常识推理：30% (150条)
    *   越界与闲聊诱导 (负样本)：20% (100条)
    *   上下文陷阱题：10% (50条)
*   **原则**：这个数据集是“考卷”，绝对不能泄露给训练集 (`train.jsonl`)。

### Step 2: 自动化评估脚本编写 (Evaluation Script)
*   **行动**：编写 `evaluate.py` 脚本。
*   **逻辑**：
    1. 加载微调后的模型。
    2. 遍历 `benchmark_v1.jsonl`，执行推理。
    3. 解析输出，对比 Ground Truth。
    4. 自动生成四项指标 (FSR, IEM, OOD-R, DCR) 的报告。

### Step 3: 数据合成与训练迭代 (Data Flywheel)
*   **行动**：如果 `evaluate.py` 输出的某项指标未达标（例如 OOD-R 只有 80%）：
    1. **归因分析**：查看模型是在哪类越界问题上犯错了（比如它能拒绝“炸弹”，但拒绝不了“写代码”）。
    2. **规则调整**：修改 `data_synthesis.py` 中的合成 Prompt，增加特定类型的负样本生成比例。
    3. **重新合成与微调**：生成新一批 `train.jsonl`，重新运行 `train.py`。
    4. **复测**：再次在 Benchmark 上跑分，直到所有指标达标。

## 📋 三、 最终验收清单 (Checklist for Production)

在将 `*.gguf` 模型推送到 Flutter 端进行集成前，必须满足以下条件：

- [ ] 模型已在最新的 `benchmark_vX.jsonl` 上完成测试。
- [ ] JSON 格式刚性解析率 (FSR) ≥ 99.5%。
- [ ] 意图精确匹配率 (IEM) ≥ 95.0%。
- [ ] 安全与越界拦截率 (OOD-R) ≥ 98.0%。
- [ ] 模型量化后的 GGUF 文件体积 < 500MB (适配 3GB 内存低端机)。
- [ ] 在 M4 (或目标端侧设备) 上的推理速度稳定在 ≥ 15 Tokens/s。

> **架构师寄语**：数据工程不是一锤子买卖，而是一个持续打磨“考卷（Benchmark）”和“教材（Train set）”的过程。守住这四大指标，我们就守住了端侧智能家居体验的底线。
