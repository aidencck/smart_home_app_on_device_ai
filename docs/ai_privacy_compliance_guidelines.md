# 智能家居 AI 系统端到端隐私合规与数据安全方案

> **文档状态**: 正式发布 (V1.0)
> **编写视角**: 隐私合规专家 (DPO / Privacy Compliance Expert)
> **适用范围**: 研发团队、产品团队、运维团队、法务团队
> **参考基准**: GDPR (欧盟), 《个人信息保护法》 (PIPL - 中国), ISO/IEC 27701 隐私信息管理体系

---

## 1. 合规核心理念：Privacy by Design (隐私保护设计)

在智能家居场景下，端云协同架构（FastAPI + 端侧 AI）天然具有保护隐私的优势，但如果在数据流转的细节中失控，依然会面临毁灭性的合规风险。本方案秉承 **“Privacy by Design”** 理念，确保合规要求内化到架构的每一行代码中，而非事后补救。

核心原则：
1. **数据最小化 (Data Minimization)**：只收集实现业务功能所绝对必需的数据。
2. **目的限制 (Purpose Limitation)**：收集的数据仅用于用户明确同意的特定目的（如：仅用于大模型微调，不用于广告推送）。
3. **默认保护 (Privacy by Default)**：在用户不进行任何设置的情况下，系统默认处于最严格的隐私保护状态（如：默认不上传日志）。

## 2. 端到端数据流转合规链路剖析

基于《FastAPI 端云协同架构落地方案》，我们将系统数据流转划分为三个高危域，并制定相应的管控措施：

### 2.1 端侧数据采集与本地存储 (Edge Domain)
**风险点**：端侧的 Isar 数据库记录了用户的所有设备行为日志（何时回家、何时关灯），这构成了高度敏感的个人生活轨迹（PII）。
*   **合规控制 1：本地存储加密**
    *   **技术要求**：Isar 数据库文件（如 `smarthome.isar`）必须启用 AES-256 全盘加密。加密密钥由手机操作系统级 Keystore / Secure Enclave 生成和托管，禁止硬编码在代码中。
*   **合规控制 2：数据生命周期管理 (TTL)**
    *   **技术要求**：端侧行为日志不能无限期保留。强制实施 30 天自动滚动删除策略。如果大模型 RAG 需要查询“去年夏天空调开了多久”，只能由云端按月聚合统计后的匿名数据提供支持，端侧不保留明细。

### 2.2 云端大模型交互 (Cloud API Domain)
**风险点**：端侧在遇到复杂指令时，会将 Query 和设备上下文 (`Context`) 发往 `/api/v1/ai/chat` 接口，可能无意中泄露用户隐私（如：“帮我提醒张三带药”）。
*   **合规控制 3：端侧前置脱敏 (On-Device Redaction)**
    *   **技术要求**：在将请求发往云端之前，Flutter 端必须运行一个极轻量的正则或 NER（命名实体识别）过滤管道。
    *   **执行标准**：将人名、电话号码、精确家庭住址替换为通用占位符。例如，将“提醒张三带药”处理为“提醒 [PERSON_1] 带药”后再发往 FastAPI 网关。
*   **合规控制 4：云端内存“阅后即焚”**
    *   **技术要求**：FastAPI 接收到的 `ChatRequest` 仅在内存中流转，完成 vLLM 或 OpenAI 推理并返回 `ChatResponse` 后，**严禁将包含用户明文指令的请求体写入任何持久化关系型数据库 (PostgreSQL) 或长期日志文件**。

### 2.3 数据飞轮与微调日志上传 (Data Flywheel Domain)
**风险点**：这是整个架构中**合规风险最高**的一环。未经授权收集“失败交互日志 (Bad Cases)”直接违反 PIPL 和 GDPR。
*   **合规控制 5：强显式授权机制 (Strict Opt-In)**
    *   **产品要求**：在 App 首次激活 AI 功能时，弹出一个单独的“智能体验改善计划”授权页面。
    *   **设计规范**：
        1. 必须有明确的《数据共享协议》链接。
        2. **“同意加入”的复选框绝对不允许默认勾选**。
        3. 必须提供一键退出（Opt-out）的开关，且退出后必须立即向云端发送“删除该设备所有历史微调数据”的指令。
*   **合规控制 6：日志差分隐私与云端清洗**
    *   **技术要求**：即使是已授权用户，端侧上传到 `/api/v1/data/telemetry` 的 Bad Cases 也必须是**脱敏+匿名化**的。
    *   **标识符剥离**：上传的日志结构中，严禁携带 `user_id`、`phone_number` 或 `device_mac_address`。必须使用定期轮换的临时会话 ID (`Session_ID`) 进行关联。
    *   **云端 LLM-as-a-Judge 的合规使命**：Celery Worker 中的大模型不仅要过滤噪音，更要承担“二次隐私审查”的职责，若发现日志中遗漏了未脱敏的 PII，直接做物理丢弃。

## 3. 商业大模型 API (OpenAI/阿里云) 的合规隔离

当 FastAPI 遇到复杂任务，决定将数据路由给第三方商业大模型时，会产生数据跨境或第三方数据共享的合规风险。

### 3.1 第三方数据共享约束
*   **企业级协议保障**：禁止使用面向 C 端的公共 OpenAI API Key。必须采购具有**“不用于训练承诺 (Zero Data Retention / No Training)”**的企业级 API 服务（如 Azure OpenAI Service）。
*   **最小化 Context 透传**：发送给第三方 API 的 System Prompt 中，只能包含解决当前 Query 绝对必需的设备状态（如查天气，就不需要附带门锁状态），严格遵循最小化原则。

## 4. 安全风控与高危指令拦截

智能家居的隐私不仅在于数据不被窃取，更在于**物理空间的安全不受侵犯**。

### 4.1 RBAC 生物认证拦截墙
*   **场景**：用户通过语音或文本要求“打开大门”、“关闭安防摄像头”。
*   **合规控制**：无论端侧模型还是云端大模型生成的 JSON 指令，只要触碰到预设的**高危设备清单 (High-Risk Device List)**，端侧 Executor 必须挂起执行流。
*   **技术动作**：唤起操作系统的 FaceID / TouchID / 指纹认证，且屏幕上必须高亮显示即将执行的高危动作，用户生物认证通过后方可执行下发。

## 5. 灾难响应与被遗忘权保障 (Right to be Forgotten)

根据合规要求，用户有权要求平台彻底抹除其所有数据痕迹。

### 5.1 一键销毁机制 (Data Eradication)
*   **产品要求**：App 设置中必须提供“注销账户并清除所有数据”的醒目按钮。
*   **后端技术动作**：
    1.  调用端侧 API 清空 Isar 数据库。
    2.  清除 PostgreSQL 中的用户账户与拓扑关系。
    3.  清除 Redis 中的设备影子。
    4.  **挑战点**：如果在微调流水线中，用户的数据已经被合成到了新版本的 GGUF / LoRA 权重中，技术上无法从权重中“剔除”单个用户的数据。
    5.  **合规声明对策**：在《隐私政策》中明确声明：“由于 AI 模型的不可逆性，您授权参与体验改善计划的数据一旦完成模型训练，将转化为不可还原的数学权重；但我们将销毁所有用于训练的原始脱敏文本及关联标识符。”

## 6. 合规审查清单 (Compliance Checklist for QA/Release)

任何 AI 相关的代码合入主干（PR/MR）及发布前，必须由 QA 团队根据以下清单进行验证：

- [ ] 端侧 Isar 数据库是否启用了全盘加密？
- [ ] 拦截 80% 流量的本地大模型，其推理过程是否完全在飞行模式（无网）下抓包验证无数据外发？
- [ ] “体验改善计划”是否做到了非默认勾选（Opt-in）？
- [ ] 发送至 FastAPI `/api/v1/ai/chat` 的 Payload 是否完成了前置的姓名/地址脱敏？
- [ ] 针对“门锁”、“摄像头”生成的控制指令，是否 100% 唤起了本地生物认证？
- [ ] FastAPI 后端数据库 (PostgreSQL) 中是否绝对没有留存用户的明文聊天记录？
- [ ] 业务方采购的外部大模型 API 凭证，是否具备“不用于模型训练”的法律条款背书？
