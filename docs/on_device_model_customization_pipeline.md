# 端侧模型深度定制与全链路微调方案 (架构师视角)

> 本文档旨在探讨如何跳出“调包侠”和“纯工程拼接”的局限，从**模型层 (Model Layer)** 深入定制专属的智能家居端侧模型。这不仅是打通端侧 Agent 全链路（Data -> Training -> Quantization -> Inference）的核心战役，更是提升团队 AI 实战技能的绝佳练兵场。

---

## 一、 为什么必须“深度定制”端侧模型？

当前开源的通用小模型（如 Qwen1.5-1.8B, Gemma-2B）虽然能力不错，但在智能家居场景下存在三个致命缺陷：
1. **领域知识稀疏**：它们更擅长写诗和闲聊，对复杂的家居联动逻辑（如“观影模式”、“离家布防”）缺乏深度理解。
2. **JSON 格式遵循度差**：即使有 GBNF 约束兜底，如果模型原生不擅长输出结构化数据，会导致采样过程频繁回退，严重拖慢生成速度 (Tokens/s)。
3. **参数冗余与体积过大**：通用知识占用了大量权重，而我们真正需要的只是一个“精准的意图翻译器”。如果能通过微调将参数量压缩到 0.5B (约 500MB)，将彻底解决低端机的内存 OOM 问题。

---

## 二、 全链路定制实战方案 (Training to Edge Pipeline)

作为架构师，我们需要带领团队搭建一套自动化的 **模型定制与下发流水线 (Model CI/CD)**。以下是具体步骤：

### 1. 数据工程：构建 SmartHome SFT 数据集
不要用网上开源的闲聊数据集。我们需要构建专门针对本项目 `AgentIntent` 结构的微调数据集。
*   **指令覆盖 (Instruction Coverage)**：
    *   直接控制指令：“把主卧的空调调到24度。”
    *   模糊意图指令：“我有点冷。” -> 需模型推理出调高温度。
    *   多意图组合：“我要睡觉了，关掉客厅灯，空调调成睡眠模式。”
    *   RAG 问答指令：基于提供的 `BehaviorLog` 上下文回答问题。
*   **数据合成策略 (Synthetic Data)**：
    利用 GPT-4 或 Claude 编写 Few-shot 脚本，基于我们 `lib/models/device.dart` 中的设备类型，批量生成 10,000+ 条高质量的对话-JSON 映射数据。
*   **格式标准 (ShareGPT Format)**：
    ```json
    {
      "messages": [
        { "role": "system", "content": "你是一个智能家居端侧Agent。当前设备列表: [...]" },
        { "role": "user", "content": "太刺眼了" },
        { "role": "assistant", "content": "{\"device_id\": \"light_1\", \"action\": \"turn_off\"}" }
      ]
    }
    ```

### 2. 模型微调：LoRA / QLoRA 极客实战
对于端侧小模型（0.5B ~ 2B），全参数微调 (Full Fine-Tuning) 成本过高且容易过拟合，**QLoRA (量化低秩微调)** 是最佳选择。
*   **基座模型选择**：推荐使用 Qwen2.5-0.5B 或 Llama-3.2-1B。它们参数极小，但底子极好。
*   **训练框架**：使用 `Unsloth` 或 `LLaMA-Factory`。它们对显存极其友好，一张单卡 RTX 4090 或甚至 Mac 的 M2 Max 即可完成训练。
*   **训练目标**：不仅要让模型学会输出 JSON，更要通过 Loss 惩罚让模型学会**“拒绝回答（拒答机制）”**——当用户问“如何制造炸弹”时，模型必须稳定输出 `{"action": "none"}`，而不是产生安全风险。

### 3. 模型量化与格式转换 (Quantization & GGUF)
训练出 LoRA 权重后，需要将其与基座模型合并，并转换为 `llama.cpp` 支持的 `.gguf` 格式。这是端侧部署的核心。
*   **格式转换**：使用 `llama.cpp` 仓库中的 `convert_hf_to_gguf.py` 脚本，将 HuggingFace 格式转换为 FP16 的 GGUF。
*   **极端量化 (Quantize)**：
    *   使用 `llama-quantize` 工具，将 FP16 模型量化为 **Q4_K_M**（4-bit 量化，中等精度，性能与体积的完美平衡）。
    *   如果目标是极低端安卓机，可以尝试 **IQ2_XXS**（2-bit 极限压缩），通过牺牲极少的精度换取巨大的内存释放。

### 4. 架构整合：动态加载与热更新 (Hot Reload for Models)
在 Flutter 架构侧，配合我们定制出的轻量级模型，实现动态更新机制：
*   **云端模型仓库 (Model Registry)**：搭建一个简单的云端配置表，记录当前最新的模型版本号、下载链接和 SHA256 校验码。
*   **后台静默下载 (Background Fetch)**：利用 `workmanager` 等插件，在手机连接 Wi-Fi 且充电时，静默下载最新的 `.gguf` 文件。
*   **热插拔引擎 (Hot Swap Engine)**：在 `LlamaCppEngine` 中实现平滑的上下文切换——在内存允许的情况下，预加载新模型，验证无误后瞬间销毁旧模型的 `llama_context`，实现用户无感的 AI 升级。

## 三、 团队技能提升与项目实战收益

将这套“模型定制全链路”引入项目，对团队的技能提升是巨大的：

1. **打破“应用层”结界**：团队成员将不再局限于写 Flutter UI 或调云端 API，而是真正深入到 `Tensor`（张量）、`Weights`（权重）和 `Quantization`（量化算法）的 AI 底层逻辑中。
2. **掌握数据飞轮 (Data Flywheel) 的精髓**：理解“模型能力上限取决于数据质量”。通过收集端侧 RAG 拦截失败的 Bad Cases，不断补充到微调数据集中，团队将实操体验到数据驱动 AI 进化的完整闭环。
3. **极致的工程性能优化经验**：在把 2B 模型塞进 3GB 可用内存的手机并跑出 20+ Tokens/s 的过程中，团队将积累极其宝贵的 C++ 内存管理、异构计算（Metal/Vulkan）和 Flutter FFI 性能调优经验。这些硬核技能是当前市场上最为稀缺的。

## 结语
定制端侧专属模型，是这个智能家居 Agent 项目从“玩具 (Toy)”走向“神器 (Killer App)”的必经之路。通过构建 Data -> Train -> Quantize -> Deploy 的自动化闭环，我们将彻底掌控 AI 的灵魂，打造出独一无二的硬核开源产品。
