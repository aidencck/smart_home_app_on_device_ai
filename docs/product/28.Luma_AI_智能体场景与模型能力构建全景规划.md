## 1. 战略定位与核心理念 (Strategic Vision)

在 Luma AI 的无感智能空间联动 (Zero-UI) 体系中，AI Agent 的定位不再是传统的“语音助手”或“指令执行器”，而是**基于多模态生理感知 (戒指/雷达) 的“环境主动编排者”**。

从产品负责人的视角来看，模型能力的构建必须服务于以下三大核心商业目标：
1. **极致的本地化与隐私保护**：生理数据（心率、HRV、血氧）高度敏感，必须在边缘侧（Edge/On-Device）完成闭环推理。
2. **100% 的控制确定性**：大模型的“幻觉”在物理世界是灾难性的（如半夜把床折叠）。必须通过工程手段（如 GBNF 语法树）将其控制意图收敛为 100% 合法的 JSON。
3. **越用越懂你的数据飞轮**：系统必须具备“被用户打断后能自我纠错”的进化能力。

## 2. AI Agent 核心应用场景矩阵 (Scenario Matrix)

基于当前的硬件生态（智能戒指、智能床、智能电视、全屋灯光），我们将 AI Agent 的场景划分为四个层级：

### 2.1 场景一：离线自然语言意图理解 (SLM 模糊控制)
*   **用户痛点**: 传统 IoT 只能听懂“打开主卧的灯并调到 50%”，听不懂“我有点冷”或“太亮了”。
*   **AI 能力要求**: 端侧小模型 (SLM) 需要将模糊的口语化表达、情绪化表达，精准映射为确定的设备控制 JSON。
*   **典型 Case**: “我要睡觉了” -> 触发多域联动（灯光淡出、电视息屏、床体放平）。

### 2.2 场景二：基于生理数据的无感编排 (Zero-UI Orchestration)
*   **用户痛点**: 睡前/睡醒需要繁琐的手动操作，打断自然节律。
*   **AI 能力要求**: 持续分析高频时间序列数据（HR、HRV、体动），进行状态跃迁的精准预测（如入睡点、深睡期、浅睡期）。
*   **典型 Case**: 
    *   **坠入梦境**: 监测到心率下降且平稳 -> 自动开启电视助眠 Shader 与白噪音 -> 床体调整为零重力。
    *   **无感唤醒**: 闹钟前 15 分钟，监测到浅睡期 -> 启动模拟日出渐变光晕与床体脉冲微震。

### 2.3 场景三：情绪补偿与生理反馈 (Affective Computing)
*   **用户痛点**: 现代人处于高压状态，环境无法提供情绪价值。
*   **AI 能力要求**: 通过 HRV (心率变异性) 评估自主神经系统状态（交感/副交感神经），识别“高压/紧绷”或“低落/疲惫”状态。
*   **典型 Case**: 识别到高压状态 -> 电视切换深海动态冷色调 -> 播放低频粉红噪音 -> 智能床开启微温热敷与波浪式摇篮起伏。

### 2.4 场景四：本地智能诊断与排障 (Local RAG)
*   **用户痛点**: 设备离线或故障时，用户不知所措。
*   **AI 能力要求**: 结合设备状态快照与本地知识库，进行故障推理与解答。
*   **典型 Case**: “为什么智能床不动了？” -> AI Agent 读取本地日志并回复“检测到深睡物理锁已开启，请先在手机端解除锁定”。

## 3. 端云协同 AI 架构设计 (Edge-Cloud Architecture)

为了支撑上述场景，我们在架构上采用**“端侧优先、云端兜底”**的混合路由（Three-Tier Routing）策略。

### 3.1 三层意图路由网关
1.  **极速规则层 (Layer 1 - 0ms)**：基于本地预设的正则表达式和白名单，直接拦截高频、简单的明确指令（如“开灯”、“关电视”）。
2.  **端侧大模型层 (Layer 2 - ~800ms)**：基于集成在 App 内的 `llama.cpp` 引擎运行的 SLM（如 Qwen2.5-0.5B 或 Llama-3.2-1B）。负责处理模糊意图、本地多设备联动和涉及敏感生理数据的场景。
3.  **云端大模型层 (Layer 3 - Fallback)**：当端侧模型置信度低、设备算力不足（电量<20% 或 RAM 紧张）或遇到需要广域知识的提问（如“明天天气”）时，平滑降级到云端 API 处理。

### 3.2 100% 确定性的指令输出 (GBNF)
针对 LLM 控制物理设备的“幻觉”痛点，我们彻底弃用了不稳定的正则提取，全面引入 **GBNF (Grammar-Based Sampling)**。
*   在边缘侧，通过预编译的 `luma_ai_control.gbnf` 语法树文件，强制 LLM 的每一个 Token 生成都必须符合 JSON-RPC 2.0 规范，以及当前全屋物模型的 Schema。
*   确保端侧解析器（Executor）接收到的永远是 100% 结构化、可执行的代码。

## 4. 模型能力构建与微调流水线 (Model Capability Build)

作为产品负责人，模型能力的建设不只是“跑通 Demo”，而是要建立一条流水线。

### 4.1 数据合成与冷启动 (Data Synthesis)
*   **策略**: 依靠开源数据集无法解决特定智能家居场景的问题。必须利用 GPT-4 等超大模型批量合成高质量的 SFT（监督微调）数据。
*   **范围**: 包含“模糊指令到确切 JSON 的映射”、“多意图组合（如：关灯并把床头抬高）”、“拒答策略（如：拒绝执行危险动作）”。

### 4.2 端侧模型极致量化 (Quantization)
*   为了确保在不同性能的手机/中控屏上流畅运行，针对低端机型采用 **IQ2_XXS (2-bit)** 极限压缩，主流机型采用 **Q4_K_M (4-bit)**。追求首字响应 (TTFT) < 300ms。

### 4.3 闭环数据飞轮与自我进化 (Data Flywheel)
这是 Luma AI 最核心的商业护城河：
1.  **冲突捕获 (Override Tracking)**：当 AI 自动执行了某个环境调节（如调暗灯光），但用户在 5 分钟内通过物理按键或 App 进行了反向操作（如重新拉亮），系统将此记录为“预测失败”的负样本 (`modified_after_execution`)。
2.  **本地脱敏存储**: 负样本及当时的生理基线数据存入本地 Isar 数据库。
3.  **云端 LLM-as-a-Judge**: 定期将脱敏后的冲突日志上传，由云端大模型进行清洗和归因分析。
4.  **端侧个性化微调 (LoRA)**: 在夜间设备闲置且充电时，利用端侧 NPU，基于清洗后的专属数据对基础模型进行微调，生成属于该用户特定睡眠习惯的 LoRA 权重，实现“千人千面”。

## 5. 里程碑与验收标准 (Roadmap & KPIs)

| 阶段 | 核心任务 | 成功标准 (产品视角 KPIs) |
| :--- | :--- | :--- |
| **Phase 1: MVP 基础设施** | 跑通 `llama.cpp` 端侧推理与 GBNF 语法树强制约束。 | 首字响应 (TTFT) < 300ms；JSON 解析成功率 100%。 |
| **Phase 2: 意图深度对齐** | 跑通数据合成流水线，覆盖 30% 核心模糊意图与 20% 拒答边界测试。 | 意图识别准确率 > 90%；离线处理占比 > 80%。 |
| **Phase 3: 闭环数据飞轮** | 上线 Override 熔断捕获机制与端云协同的 LoRA 微调。 | 用户手动干预冲突率 < 5%；AI 推荐采纳率 > 20%。 |
| **Phase 4: 生理预判管家** | 整合智能戒指数据，上线“入睡/唤醒/情绪补偿”三大主动编排场景。 | 高级订阅转化率提升；核心生理指标 (如 HRV 恢复度) 出现统计学显著改善。 |

**产品负责人结语**：
AI Agent 的成功不仅在于模型的参数量，更在于它与硬件环境（床、光、声）的深度耦合，以及对用户隐私的绝对尊重。我们将通过这条端云协同的数据飞轮，打造真正“懂你”的无感智能空间。
