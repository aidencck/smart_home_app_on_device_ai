## 🚀 Sprint 2 并行开发总线 (Parallel Tracks)
基于解耦原则，本次 Sprint 2 拆分为 **2 条完全可并行的开发主线**（前端 UI 交互与后端业务逻辑）。边缘计算层在此 Sprint 主要作为提供测试接口的配合方。

### 🧵 Track D: 云端状态与业务聚合层 (分配至 `backend-architect` 智能体)
**主线目标**: 提供 App 首页渲染所需的高并发聚合 API，以及解决多端并发控制时的状态冲突问题（Vector Clock）。
**前置依赖**: Track A (Sprint 1) 的数据库结构已就绪。

| :--- | :--- | :--- | :--- | :--- |
| **D-1** | **设计 Room 与 Scene 数据模型**<br>定义房间与预设场景（如“坠入梦境”）的关联关系。 | P0 | 1. 数据库支持场景指令组存储（如控制多个设备的指令集）。<br>2. 支持场景和用户的多对多关系。 | ✅ 已完成 |
| **D-2** | **开发首页聚合 API (`GET /v1/home/summary`)**<br>单次请求返回房间温湿度、光照、所有绑定设备状态及当前生效场景。 | P0 | 1. 避免 N+1 查询，接口响应时间 < 300ms。<br>2. 聚合数据结构符合前端 UI 渲染需求。 | ✅ 已完成 |

## 🤖 智能体 (Agents) 并行协同编排方案

### 协同策略 (Orchestration Strategy):
1.  **阶段一 (接口契约与 UI 搭建)**:
    *   唤醒 `backend-architect` 处理 **Track D** 的 D-1 和 D-2 (数据模型与聚合 API)。
    *   唤醒 `frontend-architect` 处理 **Track E** 的 E-1 和 E-4 (纯 UI 骨架与动态面板，使用 Mock 数据)。
    *(此时前后端解耦，前端专注于动画和 Riverpod 状态管理，后端专注于 SQL 优化)*

2.  **阶段二 (云端算法与端云联调)**:
    *   唤醒 `backend-architect` 处理高难度的 D-3 (Vector Clock 并发控制)。
    *   唤醒 `frontend-architect` 处理 E-2 和 E-3，将前面写好的 UI 骨架与 Track D 刚刚写好的真实 API 进行对接。
