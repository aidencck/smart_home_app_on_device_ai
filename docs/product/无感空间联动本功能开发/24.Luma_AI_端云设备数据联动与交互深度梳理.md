# Luma AI - 端云设备数据联动与交互深度梳理 (架构师视角)

**文档编号**: PRD-LUMA-DATA-024
**文档状态**: Active
**关联项目**: Luma AI 无感智能空间联动系统
**核心目的**: 作为产品负责人，深度梳理从底层硬件传感器（如智能戒指）、边缘中枢（Edge Hub/IoT Core），到云端设备影子（Device Shadow），再到用户层（App/Web UI）的数据流转协议、状态机同步机制与核心交互联动设计。**此文档指导后续的数据协议统一定义，不涉及具体代码编写。**

---

## 1. 核心挑战与现状痛点分析

当前系统的各个端（App、边缘计算设备、云端后台、真实设备）存在不同程度的数据割裂，若要实现真正的“无感智能”，必须解决以下核心矛盾：
1.  **数据颗粒度不匹配**: 
    - 硬件端（如 `ble_simulator.py`）产生的是高频、低阶的 Raw Data（如 HR=65, HRV=40, Movement=1）。
    - 边缘 AI 与 App 端需要的是低频、高阶的 Semantic Data（如 `sleep_stage: "DEEP_SLEEP"`）。
2.  **多端并发控制 (幽灵跳动)**: 
    - App 用户手动拉动滑杆、云端 AI 自动下发推荐指令、边缘自动化引擎同时触发，极易导致状态冲突与覆盖。
3.  **安全与隐私边界不清**: 
    - 生理数据（心率、睡眠）不应无限期存在云端，但设备状态（床的角度、灯光亮度）需要持久化以便恢复。

## 2. 端到端统一物模型设计 (Unified Data Schema)

为打破数据孤岛，所有设备交互必须遵循统一的 JSON 契约。

### 2.1 智能戒指 (Sensor Node: Smart Ring)
- **数据流向**: 单向 (Device -> Edge Hub -> Cloud)
- **核心数据结构 (Payload)**:
  ```json
  {
    "device_id": "ring_001",
    "timestamp": 1712400000,
    "raw_metrics": {
      "hr": 55,         // 心率 (bpm)
      "hrv": 65.5,      // 心率变异性 (ms)
      "spo2": 98,       // 血氧 (%)
      "movement": 1.2   // 体动幅度 (0-10)
    },
    "computed_state": {
      "sleep_stage": "DEEP_SLEEP" // 枚举: AWAKE, LIGHT_SLEEP, DEEP_SLEEP
    }
  }
  ```
- **联动交互规则**:
  - `raw_metrics` 仅在边缘网关 (Edge) 进行本地窗口计算，**阅后即焚**，绝不上云，保障隐私。
  - 只有 `computed_state` (如进入深睡) 作为 **Event (事件)** 向上抛出，触发空间联动。

### 2.2 智能床 (Actuator Node: Smart Bed)
- **数据流向**: 双向控制与上报 (App/Cloud <-> Device)
- **核心数据结构 (Payload)**:
  ```json
  {
    "device_id": "bed_001",
    "vector_clock": 15,
    "state": {
      "occupancy": true,        // 是否有人在床 (传感器)
      "angle": 0,               // 床头抬升角度 (0-60度)
      "vibration": "off",       // 震动状态
      "support_level": "firm",  // 支撑模式: soft, medium, firm
      "temp_mode": "auto"       // 动态微调恒温
    },
    "is_verified": false        // 二次鉴权标志位
  }
  ```
- **联动交互规则 (深睡硬锁死)**:
  - 当收到戒指的 `DEEP_SLEEP` 事件时，床体应自动将 `angle` 调至 0 (平躺)，`support_level` 调至 `firm` (护腰)。
  - **安全门禁**: 处于深睡状态时，任何非用户主动确认的 `angle` 调整请求（AI 或错误自动化）必须被后端或边缘端拦截（校验 `is_verified` 字段），防止物理形变惊醒用户。

### 2.3 智能灯光视听系统 (Environment Node: Light & TV)
- **数据流向**: 双向控制
- **核心数据结构 (Payload)**:
  ```json
  {
    "device_id": "light_001",
    "vector_clock": 42,
    "state": {
      "power": true,
      "brightness": 20,         // 亮度 0-100
      "color_temp": 2700,       // 色温 (K)
      "shader_engine": "sunrise", // 场景渲染引擎
      "transition_duration": 900  // 渐变时间 (秒)
    }
  }
  ```
- **联动交互规则 (日出渐变)**:
  - 当收到戒指的 `AWAKE` (且在晨起时间窗口) 事件时，触发 `shader_engine: "sunrise"`。
  - 系统必须下发 `transition_duration: 900` (15分钟)，由硬件固件或边缘网关执行平滑插值，禁止瞬间亮起刺眼。

## 3. 核心交互流转与仲裁机制

### 3.1 状态一致性仲裁 (Vector Clock vs Device Shadow)
解决“App 面板状态跳动”的核心在于区分**控制指令**与**状态上报**：

1. **用户主动控制 (App -> Cloud -> Device)**:
   - 采用 **乐观锁 (Vector Clock)**。
   - App 发起请求时必须携带本地缓存的 `vector_clock`。
   - 若云端版本已更新（被其他人或 AI 改了），返回 `HTTP 409 Conflict`。App 需静默拉取最新状态再重试，或提示用户。
2. **设备状态同步 (Device -> Edge -> Cloud Shadow)**:
   - 采用 **Redis Lua 脚本原子更新**。
   - 硬件按频率或状态变更上报，携带 `last_update_ts`（设备端时间戳或递增序号）。
   - 云端 Device Shadow 校验该 TS，若小于当前缓存的 TS，则视为“乱序或脏数据”直接丢弃，保证云端影子永远是最新真相。

### 3.2 人工干预与 AI 熔断 (Override & Fallback)
“无感智能”最忌讳的是“自作聪明”。

- **场景定义**: 用户处于“观影模式”（灯光暗，电视亮），但 AI 根据作息判断用户该睡觉了，自动下发关电视指令。
- **交互阻断**:
  - 用户在 App 上重新点亮电视（或调节灯光）。
  - **数据联动**: App 发出控制指令的同时，向云端并行发送一条 `Override_Log` (熔断日志，负样本)。
  - **系统响应**: 
    1. 立即挂起该房间当前运行的所有 AI 自动化（Suspend）。
    2. 将设备的 `ttl` (影子过期时间) 置为 1，强行刷新真实物理状态。
    3. 后台 AI Recommendation 引擎开始工作，次日向用户推送：“昨晚您推迟了睡眠，是否需要调整周末的作息基线？”

## 4. 商业化验收指标 (Metrics)

本数据与交互方案落地后，需通过以下数据证明价值：
1. **设备并发冲突率 (Conflict Rate)**: 目标 < 0.1%。(衡量 Vector Clock 的有效性)
2. **AI 熔断率 (Override Rate)**: 目标 < 5%。(衡量无感联动的准确度，用户越少手动调滑杆，越成功)
3. **端到端延迟 (Latency)**: 边缘闭环 < 200ms，云端闭环 < 800ms。
