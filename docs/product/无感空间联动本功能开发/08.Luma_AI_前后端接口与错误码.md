# 《前后端接口字段清单 + 状态码 + 错误码草案》

这份文档的目标，是把前面的：

* 页面
* 状态机
* 对象模型
* 前后端交互关系

进一步压成可研发落地的接口骨架。

这里不按纯技术 Swagger 写，而按 **产品负责人推动研发协同** 的方式写。
重点解决四件事：

1. 每个业务域需要哪些核心接口
2. 每个接口入参/出参的关键字段是什么
3. 状态码和业务错误码怎么分层
4. 哪些错误必须前端可理解、可转译、可落页面提示

> **架构注记（Luma AI 专属）**：由于 Luma AI 采用“端侧 Edge AI（Local Hub）+ 云端（Cloud）”协同架构，本文档定义的接口在实际落地时，部分高频、强隐私的聚合域和控制域接口会通过本地 FFI 或本地 WebSocket 承载，而配网、固件升级、长效数据备份则走云端 HTTP/MQTT。本文档统一抽象为业务接口层，研发在实现时可根据路由策略决定请求终点。

---

# 1. 文档边界与约定

## 1.1 文档范围

本稿覆盖 5 个业务域：

1. 首页聚合域
2. 设备接入与控制域
3. 场景域
4. 自动化域
5. AI 推荐与生理联动域

不覆盖：

* 登录注册
* 账号体系
* 支付订阅
* OTA 详细升级协议
* 设备底层 MQTT/长连接协议细节

---

## 1.2 接口设计原则

### 原则一：前端拿“业务可用数据”，不是拿一堆底层碎片

例如首页不应该让前端自己拼 8 个接口才知道当前房间状态。
首页必须有聚合接口。

### 原则二：原始字段和展示字段分离

例如：

* `trigger_payload` 是原始配置
* `trigger_text` 是展示文案

两者都要提供，但职责不同。

### 原则三：成功失败不只看 HTTP 状态

业务接口必须有统一业务码：

* `code`
* `message`
* `trace_id`

因为很多场景 HTTP 200，但业务失败。

### 原则四：错误必须能映射到用户提示

例如：

* 设备离线
* 场景部分执行成功
* 推荐失效
* 自动化冲突
* **生理基线获取失败（Luma 专属）**

这些都不能只给 `unknown_error`。

---

## 1.3 通用响应结构建议

### 成功响应

```json
{
  "code": 0,
  "message": "ok",
  "trace_id": "tr_123456",
  "data": {}
}
```

### 失败响应

```json
{
  "code": 200101,
  "message": "device offline",
  "trace_id": "tr_123456",
  "data": null
}
```

## 1.4 通用字段建议

所有核心接口建议统一支持：

| 字段          | 类型     | 说明                |
| ----------- | ------ | ----------------- |
| trace_id    | string | 请求链路追踪 ID         |
| timestamp   | long   | 服务端/Edge Hub 返回时间           |
| app_version | string | 客户端版本             |
| home_id     | string | 当前家庭 ID           |
| room_id     | string | 当前房间 ID           |
| request_id  | string | 客户端请求唯一 ID，适合幂等控制 |

## 2.1 获取首页聚合信息

### 接口目的

给首页一次性返回渲染所需主要信息。

### 接口建议

`GET /v1/homepage/summary`

### 请求参数

| 字段                     | 类型     | 必填 | 示例       | 说明        |
| ---------------------- | ------ | -- | -------- | --------- |
| home_id                | string | 是  | home_001 | 当前家庭      |
| room_id                | string | 是  | room_001 | 当前房间      |
| include_recommendation | bool   | 否  | true     | 是否返回推荐摘要  |
| include_automation     | bool   | 否  | true     | 是否返回自动化摘要 |

| 字段                     | 类型          | 说明       |

# 3. 设备接入与控制域接口

---

## 3.1 获取可添加设备列表（BLE 配网后确认）

严格说 BLE 扫描发生在客户端本地，但服务端通常还要做一个“设备可绑定确认”。

### 接口建议

`POST /v1/devices/provision/verify`

### 接口目的

校验设备是否可被当前用户绑定。

### 请求参数

| 字段            | 类型     | 必填 | 示例             | 说明            |
| ------------- | ------ | -- | -------------- | ------------- |
| product_id    | string | 是  | prod_light_001 | 产品 ID         |
| ble_device_id | string | 是  | ble_xxx        | BLE 扫描得到的设备标识 |
| device_token  | string | 否  | token_xxx      | 设备广播带出的绑定令牌   |
| home_id       | string | 是  | home_001       | 目标家庭          |

### 响应字段

| 字段                  | 类型     | 说明     |
| ------------------- | ------ | ------ |
| bindable            | bool   | 是否允许绑定 |
| product_name        | string | 产品名    |
| default_device_name | string | 默认设备名  |
| reason_code         | string | 不可绑定原因 |

### 响应字段

| 字段               | 类型     | 说明                   |
| ---------------- | ------ | -------------------- |
| bind_status      | string | pending/bound/failed |
| device_id        | string | 平台设备 ID              |
| online_status    | string | online/offline       |
| fail_reason_code | string | 失败原因码                |

## 3.5 房间级控制接口

### 接口建议

`POST /v1/controls/room-command`

### 接口目的

首页总开关 / 亮度 / 色温控制。

### 请求参数

| 字段            | 类型     | 必填 | 示例                | 说明      |
| ------------- | ------ | -- | ----------------- | ------- |
| request_id    | string | 是  | req_001           | 幂等请求 ID |
| home_id       | string | 是  | home_001          | 家庭 ID   |
| room_id       | string | 是  | room_001          | 房间 ID   |
| command_type  | string | 是  | set_brightness    | 指令类型    |
| command_value | object | 是  | {"brightness":28} | 指令值     |

| -------------------- | ------ | ----------------------------- |
| accepted             | bool   | 是否已受理                         |
| execution_task_id    | string | 控制任务 ID                       |
| expected_result_type | string | full_success/partial_possible |

# 7. 全局状态码与业务错误码分层设计

> **产品提示**：不要让后端直接把 500 返回给用户。产品经理必须规划一套错误码字典，让前端知道在什么情况下弹 Toast，什么情况下展示全屏空状态，什么情况下引导用户排查网络。

## 7.1 状态码设计原则

1. **HTTP 状态码保持纯粹**：网络不通给 4xx/5xx，网络通但业务失败给 HTTP 200，错误逻辑在 `code` 里体现。
2. **六位错误码分域**：
   * `0`：成功
   * `1xxxxx`：通用及鉴权域
   * `2xxxxx`：设备与 IoT 域
   * `3xxxxx`：场景与控制域
   * `4xxxxx`：自动化域
   * `5xxxxx`：AI 与边缘计算域

---

## 7.2 核心业务错误码字典

### 通用及鉴权域 (1xxxxx)

| 错误码    | 错误信息 (message)         | 前端处理建议             |
| ------ | ---------------------- | ------------------ |
| 100001 | Unauthorized           | 跳转回登录页             |
| 100002 | Home Not Found         | 提示“家庭不存在或已解散”，刷新首页 |
| 100003 | Network Timeout        | Toast: “网络请求超时，请重试” |

### 设备与 IoT 域 (2xxxxx)

| 错误码    | 错误信息 (message)               | 前端处理建议                      |
| ------ | ---------------------------- | --------------------------- |
| 200101 | Device Offline               | 提示“设备已离线”，显示排查帮助入口          |
| 200102 | Device Bind Failed           | 配网失败页展示，按具体 fail_reason 给出建议 |
| 200103 | Capability Not Supported     | 隐藏/置灰该设备的对应控制 UI（如隐藏色温滑条）   |
| 200104 | Device Firmware Too Old      | 提示“需升级固件后使用此功能”，提供升级按钮      |

### 场景与控制域 (3xxxxx)

| 错误码    | 错误信息 (message)               | 前端处理建议                         |
| ------ | ---------------------------- | ------------------------------ |
| 300101 | Scene Partial Success        | 提示“部分设备执行失败”，展示执行结果浮层          |
| 300102 | Scene Device Not Found       | 场景包含已删除的设备，提示“部分设备已移除，是否编辑场景？” |
| 300103 | Control Rate Limit Exceeded  | 用户点击过快，前端本地拦截并 Toast“操作太频繁”    |

### 自动化域 (4xxxxx)

| 错误码    | 错误信息 (message)               | 前端处理建议                    |
| ------ | ---------------------------- | ------------------------- |
| 400101 | Automation Conflict          | 创建时提示“该时间段已存在冲突的自动化规则”    |
| 400102 | Trigger Condition Invalid    | 触发条件失效（如传感器离线），在列表页标红警告   |
| 400103 | Action Target Missing        | 自动化引用的场景被删除，自动化自动变更为 disabled |

### AI 与边缘计算域 (5xxxxx) [Luma 专属]

| 错误码    | 错误信息 (message)               | 前端处理建议                            |
| ------ | ---------------------------- | --------------------------------- |
| 500101 | Edge Hub Disconnected        | 提示“本地计算网关离线，部分 AI 推荐受限”，回退到基础云端推荐 |
| 500102 | Physiological Data Missing   | 提示“需要积累 3 天数据才能生成精准建议”，展示占位骨架屏   |
| 500103 | Recommendation Expired       | 用户点击采纳时发现推荐已过期，提示“该建议已失效，为您刷新列表”  |
| 500104 | GBNF Parsing Error           | AI 生成结果不符合约束树（防幻觉拦截），前端静默丢弃该推荐卡片   |

# 8. 产品负责人结论

一份好的接口与状态码设计，是产品逻辑与研发落地之间的“防火墙”。
通过这份文档，我们确保了：

1. **首页聚合不破碎**：一个接口拿完数据，提升首屏渲染速度。
2. **场景/自动化流转清晰**：区分原始 Payload 与展示文案，前后端边界明确。
3. **AI 推荐可度量**：通过标准化的 recommendation 模型，让推荐-曝光-采纳-熔断形成完整闭环。
4. **错误兜底人性化**：通过分层的业务错误码，把冷冰冰的 500 变成用户可操作的“帮助与引导”。

接下来，研发团队可以基于这份骨架，补充具体的字段类型映射、数据库表结构（DDL）以及 Edge Hub FFI 的 C++ Header 签名，直接进入技术详设与代码开发阶段。
