# 《Luma AI 全局状态流转与端云协同交互关系（系统架构融合版）》

**文档编号**: PRD-LUMA-FLOW-007
**文档状态**: Final
**核心目的**: 颠覆传统 IoT“App-云-设备”的三层线性模型，将其升级为 Luma AI 独有的**“App - Edge Hub (本地 NPU + Isar) - 多模态设备”**闭环模型。明确在 Zero-UI 架构下，谁是状态真相、AI 预判在哪执行、手动熔断如何回传，以及系统各层面的权责边界。

---

## 1. Luma AI 系统总架构分层 (6 层重构)

在无感联动架构下，传统的 IoT 云平台被“降权”为账号管理与 OTA 通道，核心决策权下放至**边缘端 (Edge Hub)**。

### 1) 交互层 (Frontend - Flutter App)
*   **职责**: 展示 UI 状态、承接主动干预（熔断）、渲染 AI 洞察报告。
*   **技术**: Flutter Provider/Riverpod 状态管理，通过 FFI 直接与本地 C++ `unified_core` 通信。

### 2) 本地智能决策层 (Edge AI - C++ `unified_core`)
*   **职责**: **Luma AI 的大脑**。负责高频接收感知设备（戒指/雷达）数据，运行小模型进行节律预判，下发多模态指令（灯/床/电视），并进行 GBNF 安全拦截。
*   **真相源**: 这里是自动化引擎与场景解析的**唯一真相源**。

### 3) 本地数据层 (Isar Database)
*   **职责**: **Luma AI 的记忆**。存储生理基线 (Baseline)、场景配置、自动化规则、用户熔断负样本 (Override Logs)。**数据绝对不出户。**

### 4) IoT 连接与执行层 (Local LAN / BLE)
*   **职责**: 设备在局域网内接收 Edge Hub 的 JSON-RPC 指令并执行物理动作，上报当前状态（Vector Clock 解决并发跳动）。

### 5) 业务云服务层 (Cloud Services)
*   **职责被弱化**: 仅负责账号授权、家庭关系绑定、设备 OTA 固件分发、远程兜底控制（当用户不在局域网时）。

### 6) 模型训练层 (Cloud/Edge 协同)
*   **职责**: 利用夜间闲置算力，基于 Isar 中的 Override 负样本，运行 LLM-as-a-Judge 进行本地 LoRA 微调，更新决策模型权重。

## 2. 核心对象在前后端/端云的流转关系

### 2.1 状态维护真相表 (Truth Source)

| 对象类别 | UI 展示层 (Flutter) | 决策真相源 (Truth) | 同步机制 |
| :--- | :--- | :--- | :--- |
| **设备在线态** | App 渲染（灰显/亮起） | **Edge Hub 心跳检测** | SSE / WebSocket 局域网推送 |
| **环境聚合态** | App 首页控制区 | **Isar 数据库 (Room Aggregate)** | FFI 主动轮询/监听回调 |
| **场景/自动化** | App 场景页/自动化页 | **Isar 数据库** | C++ 引擎解析并执行 |
| **生理基线** | App AI 页洞察图表 | **Isar 数据库** | 戒指/雷达高频上报更新 |
| **AI 推荐生成** | App AI 推荐卡片 | **Edge AI 模型推理生成** | 生成后存入 Isar 待展示池 |

## 3. 全局状态流转图：五大核心链路

### 链路 A：主动干预与熔断链路 (The Override Flow)
*这是 Luma AI 最独特且最高频的链路。*

1.  **[触发]** 用户在 App 首页拖动亮度滑杆，或按下物理开关。
2.  **[UI 拦截]** App 立即更新 UI（乐观更新），并通过 FFI 下发控制指令给 `unified_core`。
3.  **[Edge 熔断]** C++ 引擎收到指令，判断当前房间是否处于 AI 托管态（如：正在运行“坠入梦境”）。
    *   若是，立即触发 **Override 熔断机制**，中止原节律控制脚本。
4.  **[落库]** 将用户当前操作的值（如 25% 亮度）与 AI 预判值（如 10% 亮度）打包，作为负样本写入 Isar 的 `Override_Log` 表。
5.  **[执行]** Edge Hub 向设备下发新的物理指令。
6.  **[反馈]** 设备执行完毕，上报新状态。App 更新 UI 为“手动调整（已熔断）”。

---

### 链路 B：生理触发与自动化接管链路 (Zero-UI Flow)
*这是系统替用户做事的“无感”链路。*

1.  **[感知]** 智能戒指/雷达高频上报心率与体动数据给 Edge Hub。
2.  **[决策]** C++ 引擎基于 Isar 中的 Baseline 判定用户进入“深睡期”。
3.  **[拦截]** 引擎在下发指令前，经过 GBNF 语法树校验，确保不会发出“全亮”的幻觉指令。
4.  **[执行]** 引擎向灯具下发 `[平滑息屏]` 指令，向智能床下发 `[极慢速放平]` 指令。
5.  **[聚合]** Isar 数据库更新房间聚合态：`active_scene = 深睡守护`。
6.  **[同步]** App 通过监听 FFI 数据流，将首页 UI 静默更新为“深睡守护中”。

---

### 链路 C：场景手动执行链路
1.  **[触发]** 用户点击 App 场景页的“坠入梦境”卡片。
2.  **[解析]** C++ 引擎读取 Isar 中的场景模板（包含灯光暖橘呼吸、床体 15° 零重力、电视 ASMR）。
3.  **[分发]** 引擎并发向多个设备下发指令（基于 Vector Clock 标记时序）。
4.  **[反馈]** 设备逐个响应。若某设备离线，引擎汇总为 `partial_success`。
5.  **[UI 更新]** App 弹出 Toast“坠入梦境已开启”，若部分失败则提供异常明细入口。

---

### 链路 D：AI 处方推荐与数据飞轮链路
1.  **[学习]** 夜间闲时，Edge Hub 运行轻量模型，分析当天的 Override Log 与睡眠指标。
2.  **[生成]** 生成优化处方（Recommendation），例如：“是否开启深海静默抚慰模式以降低压力？” 存入 Isar 待展示池。
3.  **[展示]** 用户次日打开 App，首页或 AI 页拉取并展示该推荐卡片。
4.  **[采纳]** 用户点击“接受”。
5.  **[闭环]** C++ 引擎根据推荐参数，自动在 Isar 中生成新的 Automation 规则，并在今晚生效。

---

### 链路 E：配网与基线初始化链路
1.  **[发现]** App 通过 BLE 发现 Edge Hub 及子设备。
2.  **[联网]** App 下发 Wi-Fi 凭证，设备连接局域网。
3.  **[云注册]** 仅进行基础的账号权属绑定（Cloud 层唯一强参与环节）。
4.  **[基线建立]** 强制引导用户佩戴感知设备，Edge Hub 开始采集前 60 秒数据，在 Isar 中建立初始 `PhysiologicalBaseline`。
5.  **[转化]** 引导体验首个多模态场景，完成 Onboarding 漏斗。

## 4. 异常流转与防御机制 (Fallback & Defensive Design)

### 4.1 幽灵跳动防御 (Vector Clock 同步异常)
*   **现象**: App 下发了关灯指令，但由于网络延迟，App 先收到了上一次心跳的“开灯”状态，导致 UI 开关回弹（幽灵跳动）。
*   **处理**: Edge Hub 维护设备的 Vector Clock。App 下发指令时带上本地时间戳，若收到的状态更新时间戳小于指令时间戳，App 拒绝渲染该状态，维持乐观更新态，直到收到最新时间戳的确认包。

### 4.2 Edge Hub 离线降级
*   **现象**: 家庭网关断电或死机。
*   **处理**: 
    *   所有“生理预判”与“复杂多模态场景”立即失效。
    *   App 首页弹出严重告警：“智能中枢离线，AI 托管已暂停”。
    *   App 切换网络通道，尝试通过云端直接向设备下发基础控制指令（仅限开关/调光，床体控制锁死以保安全）。

## 5. 产品负责人的终极控制点 (The Executive Dashboard)

在 Luma AI 的端云架构下，产品负责人不看普通的点击率，必须死盯以下 **“Zero-UI 核心指标”**：

1.  **AI 介入率 (Intervention Rate)**
    *   *定义*: 每晚用户的睡眠与唤醒周期中，由系统自动触发（非手动）的比例。
    *   *及格线*: 核心用户需达到 > 80%。
2.  **手动熔断率 (Override Rate) - 核心警报指标**
    *   *定义*: 系统介入后，用户在 15 分钟内进行手动控制打断的比例。
    *   *及格线*: 经过 14 天学习后，需降至 < 5%。若持续高于 10%，说明小模型的决策逻辑存在严重偏差。
3.  **端到端延迟达标率 (E2E Latency Compliance)**
    *   *定义*: 从生理指标变化（如心率突增）到光影/床体作出反应的时长 ≤ 800ms 的比例。这是维持“无感”体验的物理底线。
4.  **睡眠质量提升均值 (Sleep Quality Delta)**
    *   *定义*: 用户采纳 AI 处方后，入睡潜伏期 (Sleep Latency) 的平均缩短时长。
    *   *业务价值*: 这是向用户推送 SaaS 订阅服务（付费墙）时，最核心的数字凭证。

**系统总结论**：
Luma AI 的系统流转，本质上是**“边缘决策、端侧闭环、云端隐退”**的架构。
所有的状态真相（Truth）都在 Edge Hub 的 Isar 数据库中；所有的控制指令都必须经过 C++ 引擎的 GBNF 防幻觉拦截；所有的手动干预都不是 Bug，而是喂给本地联邦学习飞轮的养料。理解了这张流转图，才算真正握住了 Zero-UI 产品的系统命脉。
