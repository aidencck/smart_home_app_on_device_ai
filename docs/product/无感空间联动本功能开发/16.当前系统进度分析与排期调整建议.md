# 《Luma AI 当前系统开发进度深度分析与排期调整建议》

**基于《15.Luma_AI_项目路线图与资源排期.md》与当前代码库 (smart_home_projects) 的实勘对比**

---

## 1. 预期里程碑 vs 当前实际进度对比

根据路线图规划，项目应依次经过 **M1(方案冻结)** -> **M2(基础链路打通)** -> **M3(MVP主闭环)** -> **M4(AI与异常)** -> **M5(优化)** -> **M6(首发)**。

通过对 `fronted_project`、`backed_project` 和 `devices` 目录的深度扫描，当前项目的整体进度处于 **M2 尾声向 M3 过渡的阶段**。部分前端 UI 超前，但底层链路存在严重“空心化”风险。

### 1.1 M1: 方案定义与架构设计 (100% 完成 ✅)
*   **预期**：完成物模型、前后端接口、状态机、PRD 及排期路线图。
*   **现状**：文档库 (`docs/product/无感空间联动本功能开发/`) 体系极其完备，从宏观 IA 到最底层的 JSON-RPC 接口和错误码（如文档 `08.`）均已定稿。**M1 里程碑已高质量达成**。

### 1.2 M2: 基础链路打通 (进度约 40% 🚧 - 核心风险区)
*   **预期**：BLE 配网、设备在线状态管理、基础控灯、边缘 C++ 引擎 FFI 贯通。
*   **现状（存在严重滞后）**：
    *   **配网与绑定**：后端 (`backed_project/app/api/v1/routers/devices.py`) 仅实现了设备影子的批量同步 (`/shadow/batch`)，缺失最核心的 `POST /v1/devices/bind` 配网与绑定逻辑。前端虽然有 `GatewayIntegrationPage`，但没有真实的 BLE 发现协议。
    *   **边缘 AI 引擎 (`unified_core`)**：C++ 源码 (`devices/unified_core.cpp`) 已经建立了 FFI 接口规范，但**内部全是硬编码的 Mock 数据**。例如 `evaluate_intent_complexity` 仅靠简单的字符串匹配（搜索“如果”）来判断意图，尚未真正集成轻量级 LLM 或 GBNF 语法树校验。

### 1.3 M3: 产品骨架与主流程闭环 (进度约 50% 🚧 - 前端超前)
*   **预期**：App 4大核心页面（首页、场景、自动化、AI）联调，Isar 本地库集成。
*   **现状**：
    *   **Flutter UI 骨架 (100% 完成)**：`home_shell.dart` 已经搭起了完美的 5 Tab 结构，首页 (`AiAgentDemoPage`)、场景 (`ScenesPage`)、自动化 (`AutomationsPage`) 均已开发完毕。
    *   **Isar 本地库 (100% 集成)**：`pubspec.yaml` 已深度集成 Isar 3.1.0，且在行为日志 (`BehaviorLog`) 中已投入使用，这为后续的“生理基线”存储打下了极好的底座。
    *   **业务逻辑闭环 (0% 完成)**：UI 上的点击（如触发“坠入梦境”场景）目前大多是 `SnackBar` 的前端模拟提示，并未真正下发指令到网关执行。

## 2. 当前系统架构的“断层”诊断

站在产品负责人的视角，当前项目存在一个非常典型的**“哑铃型”研发结构**：
*   **两头重**：顶层的文档规划极其清晰，前端 App 的 UI 还原度极高。
*   **中间空**：最致命的中间层——**Edge Hub (边缘网关) 的控制链路与 AI 推理能力处于缺失状态**。

如果按目前的惯性继续往下做（继续丰富 App UI），项目将面临在 M4 阶段“全面崩盘”的风险，因为底层根本撑不住 Zero-UI 的 800ms 延迟和防幻觉要求。

---

## 3. 下一步研发排期与资源调整建议 (Action Items)

为了确保路线图不失控，必须立即**暂停前端 UI 的新增开发**，将所有核心资源（特别是后端和 C++ 研发）压到以下三个 P0 级任务上：

### 🚨 Action 1：补齐 M2 欠债，打通真实配网与绑定链路 (建议周期：2 周)
*   **后端任务**：立即在 FastAPI 中实现文档 `08.` 中定义的设备绑定 (`bind`)、解绑 (`unbind`) 及心跳在线状态 (`status`) 接口。
*   **设备端任务**：提供至少 1 个智能灯具或智能床的真实 BLE 广播模拟器，让 App 能真实扫到并完成配网闭环。
*   **判断标准**：设备能在 App 中出现，且后端数据库能查到绑定关系。

### 🚨 Action 2：做实 `unified_core` 边缘引擎 (建议周期：3 周)
*   **C++ 开发任务**：废弃 `unified_core.cpp` 中的字符串 Mock 逻辑。必须引入轻量级推理框架（如 `llama.cpp` 的精简版），并实现 **GBNF (Grammar-Based Network Format)** 语法校验器，确保生成的 JSON 指令格式严格合法（防幻觉护栏）。
*   **FFI 联调任务**：Flutter 前端将真实的滑杆拖动数据（Brightness, Color Temp）通过 FFI 传入 C++ 引擎，引擎立刻返回合法的 JSON-RPC 动作指令。

### 🚨 Action 3：Isar 数据库与“生理基线”打通 (建议周期：2 周)
*   **前端/数据任务**：基于已有的 Isar 框架，创建 `PhysiologicalBaseline` 表和 `Override_Log`（熔断负样本）表。
*   **联调任务**：当用户在首页（已运行场景时）手动调节亮度，必须拦截该操作，并写入一条 `Override_Log` 到 Isar 中，作为未来联邦学习的飞轮数据。

---

## 4. 结论

当前项目规划宏大且逻辑严密，前端进展令人惊喜。但必须正视**“边缘计算层严重 Mock 化”**的现实。

**路线图调整决定**：原定 M3（主闭环联调）的时间节点必须往后推迟 **2-3 周**，以换取 M2（基础设备链路与 C++ 引擎）的真正做实。只有把中间层做扎实，Luma AI 才不会沦为一个漂亮的 PPT 演示 App。
