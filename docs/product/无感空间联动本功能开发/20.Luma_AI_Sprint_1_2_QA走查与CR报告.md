## 🔍 第一部分：Code Review 深度诊断报告 (架构师视角)

经过对底层代码的静态扫描与架构分析，我们在当前版本中发现了几个**可能在百万级设备规模下导致数据库宕机或数据错乱**的严重问题。

### 🚨 1. 致命的数据库性能瓶颈 (N+1 与全表扫描)

- **问题描述**:
  - `Device` 模型中的 `is_online` 和 `last_seen` 字段**缺少数据库索引**。
  - `offline_worker.py` 中的定时任务每分钟都会执行一次 `UPDATE` 检查离线设备，由于没有索引，这会导致**全表扫描 (Full Table Scan)**。设备量级增加时，将耗尽数据库 CPU 并引发锁升级（行锁变表锁），阻塞整个系统。
- **修复方案**:
  在 `app/models/device.py` 中为 `Device` 添加专用的复合索引：
  ```python
  __table_args__ = (Index('ix_device_online_status', 'is_online', 'last_seen'),)
  ```

### 🚨 2. 高危的并发数据污染 (设备重复绑定)

- **问题描述**:
  `DeviceService.bind_device` 接口在插入数据前使用了 `SELECT` 检查是否已绑定，但未加锁。在弱网环境用户狂点或并发请求下，会出现 **TOCTOU (Time-of-Check to Time-of-Use)** 竞态条件，导致同一个用户对同一个设备生成多条相同的绑定记录。
- **修复方案**:
  在 `Binding` 模型中添加联合唯一约束，并在 API 层捕获 `IntegrityError` 抛出 HTTP 409。
  ```python
  UniqueConstraint('user_id', 'device_id', name='uq_user_device_binding')
  ```

### ⚠️ 3. Vector Clock 机制未闭环

- **问题描述**:
  - 状态更新接口 (`PUT /v1/devices/{id}/state`) 虽然校验了 Vector Clock，但在成功更新后，**未将新的版本号返回给客户端**。这导致客户端下次更新时必然因为拿着旧版本号而触发 409 冲突。
  - Lua 脚本中错误地使用了物理时间戳 (`req_ts`) 作为防并发依据，这极易受到设备时钟漂移 (Clock Skew) 攻击。
- **修复方案**:
  修改 API 返回值，闭环时钟：
  ```python
  return BaseResponse.success(data={"vector_clock": new_clock})
  ```

### ⚠️ 4. 后台任务缺少优雅停机 (Graceful Shutdown)

- **问题描述**:
  `start_offline_worker` 里的 `while True:` 循环没有捕获 `asyncio.CancelledError`。当 FastAPI 重启或关闭时，容易产生僵尸进程或强制截断数据库事务。

## 🧪 第二部分：QA 全链路走查测试用例 (执行清单)

针对 Sprint 1 & 2 开发的接口，QA 团队需要手动或通过自动化脚本 (如 Pytest/Postman) 验证以下链路：

### 🟢 1. 基础设备生命周期链路

- [ ] **用例 1.1**: App 发起 `POST /v1/devices/bind`，传入合法的 MAC 和设备类型。预期返回 200，并在 DB 中生成 Owner 角色的绑定关系。
- [ ] **用例 1.2**: 设备端每隔 60 秒发起 `POST /v1/devices/heartbeat`。预期数据库的 `last_seen` 更新，`is_online` 变为 True。
- [ ] **用例 1.3**: 停止发送心跳 3 分钟以上，观察 `offline_worker` 是否成功将该设备的 `is_online` 变更为 False。

### 🟡 2. 高并发与防跳动链路 (Vector Clock)

- [x] **用例 2.1**: App A 和 App B 同时拉取设备状态，拿到相同的 `vector_clock = 5`。
- [ ] **用例 2.2**: App A 发起 `PUT /v1/devices/{id}/state`，将亮度改为 80，带上 `clock=5`。预期返回 200，并下发新的 `clock=6`。
- [ ] **用例 2.3**: 此时 App B 发起 `PUT /v1/devices/{id}/state`，将亮度改为 20，带上旧的 `clock=5`。**预期被后端严格拦截，返回 409 Conflict 异常**，确保了状态不会被幽灵覆盖。

### 🟢 3. 首页聚合性能链路

- [ ] **用例 3.1**: 调用 `GET /v1/home/summary`。预期返回完整的房间、设备列表及当前生效场景。
- [ ] **用例 3.2**: 开启 SQL 打印日志，验证该接口的底层查询是否被优化为了 3 条平行的 Join 查询，**绝不能出现 N+1 查询风暴**。
