# 智能家居 App：端侧 AI Agent 技术产品方案预研

## 1. 背景与核心价值
随着大模型（LLM）的小型化和移动端算力的提升，将 AI Agent 部署在端侧（手机/平板/智能中控）成为智能家居领域的新趋势。
相比于传统的云端 AI，端侧 AI 具有以下不可替代的核心价值：
- **极致隐私保护**：家庭内部对话、作息习惯、设备状态等敏感数据无需上传云端，完全在本地闭环处理。
- **无网/弱网可用（Offline-first）**：即使在断网情况下，用户依然可以通过语音指令或自然语言控制家电，保证智能家居的基础体验。
- **超低延迟**：省去云端网络请求的 RTT（往返时间），指令下发到设备响应的延迟可以控制在毫秒级。
- **零云端算力成本**：随着用户量增加，无需承担高昂的云端推理 API 费用。

## 2. 端侧 AI Agent 核心应用场景
### 2.1 离线自然语言控制与意图理解
- **传统方式**：必须说出固定的唤醒词和指令模板（如“打开客厅的灯”）。
- **端侧 Agent**：支持模糊意图。例如用户说“我有点冷”或者“光线太暗了”，端侧 SLM（小语言模型）能够结合当前环境温度和光照传感器数据，自动推断出“调高空调温度”或“打开主灯”的执行策略。

### 2.2 个人习惯的本地学习（Proactive Agent）
- 驻留在端侧的 Agent 可以在后台持续记录用户的设备使用习惯（如每天晚上 11 点关灯、调低空调风速）。
- 基于本地历史数据，Agent 可以主动构建自动化规则，并在合适的时机弹出建议：“发现您通常在这个时间休息，是否为您开启睡眠模式？”。

### 2.3 家庭设备异常诊断助手
- 当某个智能设备离线或连接失败时，用户可以直接询问 App：“为什么扫地机器人不动了？”
- 端侧 Agent 可以读取本地的设备日志和网络状态，结合本地 RAG（检索增强生成）知识库，给出具体的排查建议（如“检测到 Wi-Fi 信号弱”或“主刷被异物卡住”）。

## 3. 技术架构选型与落地路径 (Flutter 环境)

本方案基于现有的 Flutter 智能家居项目 (`smart_home_app`)，提出以下技术栈预研：

### 3.1 核心组件层选型
| 功能模块 | 推荐技术栈/框架 | 说明与优势 |
| :--- | :--- | :--- |
| **端侧大语言模型 (SLM)** | Google Gemma-2B / Qwen-1.5-1.8B / Llama-3 (量化版) | 参数量在 1B-3B 之间，适合在高端 Android/iOS 设备上以 4-bit 量化运行。 |
| **推理引擎** | **MediaPipe LLM Inference API** 或 **llama.cpp** | MediaPipe 提供了官方的跨平台支持；llama.cpp 对 Apple Silicon (Metal) 和 Android (OpenCL) 有极致优化。 |
| **本地语音识别 (ASR)** | Whisper.cpp (Tiny/Base 模型) / 苹果官方 Speech 框架 | Whisper.cpp 支持完全离线高精度转写，Flutter 可通过 ffi 调用。 |
| **本地语音合成 (TTS)** | flutter_tts (系统自带) / VITS 端侧模型 | 优先使用系统原生 TTS 以节省内存。 |

### 3.2 Flutter 架构集成方案
1. **感知层 (Perception)**: 
   - 收集语音输入转文本。
   - 收集当前家庭设备的 Context（如：当前哪些灯亮着、温湿度是多少）。
2. **大脑层 (Brain - 端侧 SLM)**: 
   - 将 Context 和 User Query 组合成 Prompt 喂给端侧量化模型。
   - **Prompt 示例**: `你是一个智能家居助手。当前客厅灯:关, 空调:24度。用户说:"我有点热"。请输出需要执行的 JSON 指令。`
   - 要求模型输出结构化的 JSON (Function Calling 思想)，例如：`{"device": "air_conditioner", "action": "set_temperature", "value": 22}`。
3. **执行层 (Execution)**: 
   - App 解析生成的 JSON，通过本地局域网协议（如 Matter, Thread, 本地 MQTT）直接向智能设备发送控制指令。

## 4. 面临的挑战与应对策略
1. **包体积与内存占用过大**：
   - *策略*：模型文件（通常 >1GB）不打包在安装包内，而是在用户首次使用 Agent 功能时按需动态下载（On-Demand Download）。运行时采用 mmap 机制加载模型以优化内存峰值。
2. **低端机型算力不足**：
   - *策略*：采取**端云协同**架构。检测到设备算力不足或电量低于 20% 时，无缝降级到云端大模型 API 处理；在高端机型且电量充足时，优先使用端侧推理。
3. **模型幻觉（输出非法的控制指令）**：
   - *策略*：在执行层加入**硬编码的安全校验拦截器**。例如模型要求“将空调设置为 -10 度”时，拦截器检测到超出合理范围（16-30度），则拒绝执行并提示用户。

## 5. 产品与技术指标体系 (Metrics & KPIs)
为了科学评估端侧 AI Agent 的表现和用户体验，我们建立了以下产品与技术指标体系：

### 5.1 产品体验指标 (Product Metrics)
- **任务完成率 (Task Success Rate)**：用户通过自然语言发出的设备控制指令中，被 Agent 正确解析并执行成功的比例（目标 > 90%）。
- **意图识别准确率 (Intent Accuracy)**：端侧模型对模糊指令（如“太冷了”）推断出正确动作的准确率。
- **离线处理占比 (Offline Handled Rate)**：在所有 Agent 交互中，完全在端侧本地闭环处理完成的比例（目标 > 80%），体现无网可用性和对云端成本的节省。
- **功能活跃度 (Agent DAU/MAU)**：智能家居 App 日活用户中，使用语音或文本呼出 AI Agent 功能的用户占比。
- **主动推荐采纳率 (Proactive Acceptance Rate)**：端侧 Agent 根据用户习惯主动弹出的控制建议被用户点击确认的比例。

### 5.2 技术性能指标 (Technical Metrics)
- **端到端延迟 (End-to-End Latency)**：从用户语音说完/点击发送，到设备实际开始响应的耗时（包含 ASR、推理、指令下发，目标 < 800ms）。
- **首字响应时间 (TTFT - Time To First Token)**：端侧大模型开始输出第一个字符的时间（目标 < 300ms）。
- **推理速度 (TPS - Tokens Per Second)**：端侧模型在主流机型（如 iPhone 13 / 骁龙 8 Gen 1）上的生成速度（目标 > 15 token/s）。
- **内存峰值占用 (Peak Memory Usage)**：模型加载与推理期间，App 的额外内存占用量（对于 2B 级别模型，需控制在 1.5GB - 2GB 以内，避免触发 OOM）。
- **耗电量 (Battery Consumption)**：Agent 唤醒和推理过程对手机电池电量的消耗情况（单次交互耗电量应极低）。
- **冷启动与加载耗时 (Model Load Time)**：App 首次冷启动时，从磁盘加载量化模型到内存所需的耗时（结合 mmap 机制，目标 < 2s）。

## 6. 演进路线规划 (Roadmap)
- **Phase 1: 基础设施搭建 (MVP)**
  - 集成 `llama.cpp` 或 `MediaPipe` 到 Flutter 项目。
  - 完成端侧模型的下载、加载与基础对话测试。
- **Phase 2: 意图控制闭环**
  - 设计智能家居状态的 Context 注入机制。
  - 训练或微调适用于智能家居指令解析的小模型（LoRA 微调）。
  - 实现从自然语言到设备控制 API 的离线闭环。
- **Phase 3: 主动智能与语音唤醒**
  - 加入端侧 Whisper 语音唤醒与离线语音识别。
  - 结合时间、地理位置、设备状态日志，实现预测性的主动推荐功能。
