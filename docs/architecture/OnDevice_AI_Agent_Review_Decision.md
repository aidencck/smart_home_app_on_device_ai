# 智能家居端侧 AI Agent 方案复盘与决策文档

## 1. 方案背景与初衷

随着智能家居设备的普及，用户对控制体验的要求日益提高。传统的云端 AI 助手（如天猫精灵、小爱同学）虽然能力强大，但在**隐私保护**、**断网可用性**以及**响应延迟**方面存在固有的局限性。

本项目旨在探索并落地一套基于 **Flutter + 端侧大模型 (SLM)** 的智能家居 AI Agent 方案。通过在移动端（手机/平板）直接运行量化后的大模型，实现完全离线的自然语言意图理解和设备控制。

### 核心价值驱动：
*   **极致隐私**：用户的作息、设备状态、对话内容均不出端，从根本上解决隐私焦虑。
*   **离线可用 (Offline-first)**：即使家庭断网，只要局域网通畅，用户仍可通过自然语言控制设备。
*   **超低延迟**：省去云端网络 RTT，实现毫秒级的指令响应。
*   **零云端算力成本**：边际成本递减，无需为用户的高频指令支付昂贵的云端 API 费用。

## 2. 现有方案技术架构评审

目前的方案基于 Clean Architecture 思想进行了良好的模块化拆解，核心模块包括：

1.  **推理引擎 (`agent_inference_engine`)**：基于 `llama.cpp`，通过 FFI 与 Dart 通信，支持 iOS (Metal) 和 Android (OpenCL) 的硬件加速。
2.  **上下文感知 (`agent_context_provider`)**：负责收集当前设备状态，并将其序列化为极简格式注入 Prompt。
3.  **执行与安全 (`agent_action_executor`)**：负责解析模型输出的 JSON，进行 Schema 校验和边界值拦截。
4.  **交互 UI (`agent_ui_components`)**：提供了流式的聊天界面、快捷指令和设备卡片的富文本展示。

### 2.1 架构亮点 (The Good)
*   **模块解耦清晰**：推理、上下文、执行三个核心模块职责分明，便于独立测试和后续替换（如将 `llama.cpp` 替换为 `MediaPipe`）。
*   **多轮对话支持**：`AgentContextProvider` 引入了历史消息缓存（最近 10 条），使得模型具备了基本的指代消解能力（如“关掉它”、“调高一点”）。
*   **UI/UX 融合度高**：创新的将设备控制卡片 (`DeviceCard`) 直接嵌入聊天流中，实现了“对话即控制，控制即反馈”的直观体验。并且打通了全局的状态管理 (`DeviceManager`)，保持了应用各处状态的一致性。
*   **GBNF 语法约束**：在调用推理引擎时加入了 JSON 语法约束 (`grammarSchema`)，极大降低了小模型产生幻觉或输出非法格式的概率。

### 2.2 存在的短板与风险 (The Bad & The Ugly)
*   **模型体积与内存墙 (OOM)**：即便是量化后的 2B 模型，体积仍在 1.5GB 左右。在低端机型或可用内存 < 2GB 的设备上极易触发 OOM 崩溃。目前缺乏完善的内存检测和模型动态卸载机制。
*   **耗电与发热**：端侧推理是计算密集型任务。如果用户进行长篇大论的闲聊，会导致手机严重发热并触发降频。
*   **冷启动延迟**：首次加载 1.5GB 的模型到内存（即便使用 mmap）仍需要数秒时间，这期间用户的输入会被阻塞，体验不佳。
*   **IoT 协议对接缺失**：目前的执行层 (`agent_action_executor`) 仅停留在 UI 状态的 Mock 阶段，尚未真正打通 MQTT、Matter 或 HomeKit 等真实的本地局域网控制协议。

## 3. 核心功能演进复盘 (2023-10 更新)

在近期的迭代中，我们对 Agent 界面进行了深度的体验优化，彻底补齐了之前遗留的体验短板：

1.  **从单点控制到场景联动**：支持“看电影”、“出门模式”等多设备联动意图，并在 UI 上支持横向滑动的**多卡片聚合展示**。
2.  **上下文意图继承**：通过动态注入历史对话上下文，成功实现了“有点冷 -> 开空调”到“再调高一点 -> 空调调至 28 度”的连贯交互。
3.  **内存监控与防崩溃 (OOM Prevention)**：
    *   在 `SmartHomeAgent.initialize` 阶段引入了设备内存状态的探测。
    *   当识别到低内存设备时，提供明确的 UI 预警提示，并为后续的模型动态卸载或云端路由降级打下基础，大幅提升系统健壮性。
4.  **全链路免提语音体验 (Voice-First)**：
    *   打通了基于端侧 ASR（模拟 Whisper）的语音录入、意图识别与自动发送闭环。
    *   UI 层面增加了沉浸式的“🎙️ 正在聆听”气泡与加载动效，让“动嘴不动手”的交互自然顺畅。
5.  **主动式管家引擎 (Proactive Recommendation)**：
    *   从“被动响应”跨越至“主动服务”。
    *   基于时间、环境参数（Mock）的本地规则引擎，能在适当时机（如深夜或傍晚）主动推送带有“一键执行”按钮的特殊 UI 卡片（如“睡眠模式”或“回家模式”）。
6.  **思维链与性能透明化**：在开发者模式下，UI 会展示模型推理的“思维链 (Processing Steps)”和性能指标（如 Tokens/s），极大地便利了后期的评测与调优。

## 4. 产品与技术决策点 (Decision Points)

基于当前已经非常丰满的端侧体验，团队在推进下一阶段（V2.0 商业化落地）时，需要做出以下战略级决策：

### 决策 1：纯端侧 vs 端云协同路由 (Hybrid Routing)
*   **现状**：目前端侧逻辑和兜底（Fallback）已非常完善，但面对需要外部知识（如：“今天出门需要带伞吗？并帮我把空调关了”）时依然无力。
*   **✅ 决策结论：引入“端云协同路由器 (Semantic Router)”。**
    *   **原则**：设备控制、隐私查询、本地化规则推荐必须 100% 在端侧完成。
    *   **落地**：在意图解析层前置一个轻量级的分类器（可以是本地小模型或规则网络），当判定指令属于“外部知识”、“复杂推理”或“当前设备低电量/低内存”时，无缝切换至云端大模型 API（如 GPT-4o 或通义千问）。UI 上给予用户明确的“正在请求云端”的标识以保障知情权。

### 决策 2：模型动态分发与热更新策略
*   **现状**：应用包无法直接内置 1.5GB 的 GGUF 模型文件。
*   **✅ 决策结论：构建“模型基座分发平台 (Model Hub)”。**
    *   实施**按需下载 (On-Demand)**：用户首次开启 Agent 功能时，在 Wi-Fi 环境下静默下载。
    *   **设备分级 (Device Profiling)**：根据 iOS/Android 设备的 CPU/NPU 跑分和内存大小，下发不同参数量和量化精度的模型（如旗舰机下发 Q8_0，低端机下发 Q4_K_M 甚至仅保留云端能力）。
    *   **增量更新**：模型的微调权重（LoRA）可以通过极小的包体积进行热更新，以不断学习用户的新习惯。

### 决策 3：商业化生态与协议融合
*   **现状**：目前的 `DeviceManager` 能够很好地管理 UI 状态，但与底层硬件处于隔离状态。
*   **✅ 决策结论：全面拥抱 Matter 标准，打造“生态连接器”。**
    *   Agent 侧输出的标准 JSON (`AgentIntent`) 必须直接映射为 Matter 协议的 Cluster Commands。
    *   这意味着我们的 App 不仅能控制自家的智能设备，还能无缝接管支持 Matter 的第三方设备（如飞利浦 Hue、绿米等），将 Agent 的价值最大化，成为真正的“全屋智能大脑”。

### 决策 4：从“规则引擎”向“端侧持续学习 (Local RAG / LoRA)”演进
*   **现状**：目前的主动推荐引擎是基于硬编码（Hard-coded）规则的（例如：晚上10点推睡眠模式）。
*   **✅ 决策结论：启动端侧行为日志向量化项目。**
    *   利用现有的 Isar 本地数据库，记录用户的操作习惯序列。
    *   引入端侧的小型 Embedding 模型，将用户习惯向量化，通过 Local RAG 技术，让 Agent 真正做到“千人千面”。例如，Agent 会学习到“该用户喜欢在室温高于 28 度且打开电视时，自动将空调调至 24 度”。

## 5. 下一步商业化行动计划 (Next Steps to Go-To-Market)

1.  **P0 - 云端协同基建**：开发轻量级 Semantic Router，完成云端大模型 API 的安全接入与降级机制。
2.  **P0 - 模型分发基建**：搭建后端的模型文件存储桶与客户端的断点续传/校验逻辑。
3.  **P1 - 真实 IoT 链路打通**：将 `AgentActionExecutor` 与真实的 MQTT Broker 或 Matter Controller SDK 桥接，替换掉当前的 Mock 状态。
4.  **P2 - 本地 RAG 原型验证**：基于现有的 `BehaviorLog` 探索基于端侧向量检索的个性化推荐算法。
