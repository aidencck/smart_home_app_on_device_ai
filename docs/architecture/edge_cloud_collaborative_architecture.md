# 智能家居端云协同 AI 架构设计方案

## 1. 架构概述与核心理念

当前智能家居系统的 AI 演进面临隐私泄露风险、网络延迟高、断网不可用等行业痛点。本项目秉承**“端侧优先、云端兜底、协同进化”**的核心理念，构建了一套高度鲁棒的端云协同 AI 架构。

*   **极致隐私保护**：家庭敏感数据（作息习惯、设备状态、私密对话）默认不出局域网，端侧闭环处理。
*   **无网可用 (Offline-First)**：确保断网状态下，基础意图推断和设备控制依然流畅不瘫痪。
*   **端云自适应调度**：根据端侧硬件算力、系统可用内存和电池电量，动态决定推理负载在端侧还是云端的分配。

---

## 2. 端侧 AI (On-Device AI) 深度剖析

项目基于 `packages/on_device_agent` 实现了生产级的端侧大模型推理能力，成功将 AI Agent 落地于移动端，其核心技术剖析如下：

### 2.1 异步隔离与高性能推理引擎
*   **底层引擎**：基于 `llama.cpp`，并利用 Dart FFI (Foreign Function Interface) 实现 Flutter 与底层 C++ 的高效通信。
*   **Isolate 隔离**：将模型的加载 (Mmap)、Prompt 评估和 Token 生成等重度 CPU/GPU 计算全部置于 Dart 的后台 **Isolate** 中执行，绝对不阻塞 Flutter Main UI 线程，确保交互和动画维持 60fps 流畅度。
*   **内存防爆**：强制启用 `mmap`（内存映射），由操作系统接管内存换页，避免 2B 级量化模型（约 1.5GB）直接吃满 RAM 触发 OOM。

### 2.2 零幻觉的确定性控制 (Deterministic Execution)
*   **动态 GBNF (Grammar-Based Sampling)**：废弃了不可靠的正则表达式提取方式。每次推理前，系统向状态管理层获取当前真实存在的设备列表，动态生成 GBNF 语法树（如 `device_id ::= "\"light_1\"" | "\"ac_1\""`）并注入到采样器中。
*   在 Logits 采样阶段直接掐断非法字符的生成可能，实现了 **100% 的 JSON 结构和物理设备实体解析准确率**。

### 2.3 端侧极轻量知识库 (Edge RAG)
*   摒弃了高功耗、笨重的本地向量数据库，采用高性能本地对象数据库 **Isar** 存储设备的 `BehaviorLog`。
*   **上下文动态裁剪**：系统通过轻量级意图路由拦截查询指令，将 Isar 中的近期行为日志转化为自然语言，作为 Context 动态注入 System Prompt，赋予端侧大模型本地“记忆”（例如回答“昨天空调开了多久”）。

## 3. 端云协同架构设计方案 (Edge-Cloud Collaboration)

在夯实端侧 AI 的基础上，为了解决硬件算力天花板和跨域泛化能力不足的问题，本项目设计了以下端云协同机制：

### 3.1 三层意图路由网关 (Three-Tier Intent Routing Gateway)
系统通过多层级网关，将自然语言指令动态路由到最合适的处理引擎：

1.  **Layer 1：极速规则层 (延迟 0ms)**
    *   **职责**：处理高频、明确的简单控制（如“开灯”、“关窗帘”）。
    *   **机制**：基于纯正则和关键词拦截，直接映射并下发局域网 IoT 控制指令。
2.  **Layer 2：端侧大模型层 (延迟 ~800ms)**
    *   **职责**：处理模糊意图（如“我有点冷”）、本地多设备联动、本地 RAG 数据查询。
    *   **机制**：唤起端侧量化模型（如 Gemma-2B / Llama-3 4-bit），结合 GBNF 生成执行 JSON。
3.  **Layer 3：云端大模型层 (Cloud Fallback)**
    *   **职责**：处理非家居域指令（Out of Domain，如“今天天气如何”、“帮我写封邮件”）、复杂的规划推理。
    *   **机制**：将 Query 透传给云端大参数模型（如 GPT-4o / 阿里云百炼），作为能力兜底。

### 3.2 动态算力自适应与无缝降级 (Adaptive Compute Scheduling)
架构具备**硬件感知探针**能力，当满足以下任一极端条件时，系统自动挂起端侧推理，将请求无缝降级至云端 API（云端部署同款模型架构以保证输出格式一致）：
*   **内存探针报警**：检测到系统当前可用 RAM < 2.5GB，存在较高 OOM 崩溃风险。
*   **电量与发热保护**：手机电量 < 20%，或触发操作系统级高温降频（Thermal Throttling）。

### 3.3 模型生命周期与 OTA 动态分发 (Model OTA & Lifecycle)
解决 App 包体积膨胀（通常 > 1GB）的问题：
*   **初始包分离**：Flutter 应用安装包（APK/IPA）内不内置 GGUF 模型文件，控制在 50MB 以内。
*   **按需动态下发**：用户首次进入 Agent 界面时，触发后台下载管理器，支持断点续传与 MD5 校验。
*   **千机千面策略**：启动时检测 NPU 算力和 RAM 大小，高端机（>8GB RAM）下发 3B 模型，中低端机（<4GB RAM）下发 0.5B 模型或默认切换至云端。

### 3.4 端云数据飞轮与持续进化 (Data Flywheel via Model Forge)
基于 `model_forge` 模块构建从数据合成到模型更新的闭环：
1.  **端侧脱敏上传**：用户在端侧的失败指令交互和高频特定习惯，经严格脱敏与匿名化后同步至云端。
2.  **云端微调 (Fine-tuning)**：利用 `model_forge/notebooks/data_synthesis.py` 扩充高质量数据集，使用 Apple MLX (或云端 GPU) 对模型进行 LoRA 微调。
3.  **权重下发**：将微调后体积仅数 MB 的 LoRA 适配器（Adapters）通过 OTA 下发至端侧并与基座模型合并，实现端侧体验“越用越聪明”。

## 4. 关键系统交互链路 (Key Interaction Flow)

1. **用户输入** (语音 ASR 转文本 / 键盘输入) -> 送入 **意图网关 (Intent Router)**。
2. 若为明确指令 -> **Layer 1 规则引擎** -> 解析完毕。
3. 若需模型介入 -> **硬件探针检测**：
   *   资源充足 -> 读取 Isar 日志拼接 Context -> **端侧 Isolate (Layer 2) + GBNF 推理**。
   *   资源受限/域外问题 -> **云端模型 (Layer 3) 请求**。
4. 模型输出 JSON -> 进入 **执行与安全护栏 (Action Executor & Guardrails)**。
   *   *边界校验*：拒绝非法的参数（如空调设置 -10 度）。
   *   *RBAC 生物认证*：若控制门锁、摄像头等高危设备，挂起执行流，唤起系统 FaceID/TouchID，验证通过后放行。
5. 通过局域网（Matter/MQTT）下发硬件指令。
6. 指令成功后，将结果异步写入本地 **Isar 数据库**，并在 UI 渲染状态对比卡片与思维链。

## 5. 核心性能与体验指标 (Performance Metrics)

*   **延迟指标**：首字响应时间 (TTFT) 控制在 < 300ms，端到端控制总延迟 < 800ms。
*   **准确率指标**：意图识别及 JSON 格式合法率达到 100%（归功于 GBNF）。
*   **稳定性指标**：0 OOM 闪退率，单次交互耗电量极低，保障长尾运行的稳定性。
