# 智能家居端侧 AI Agent 架构复盘与演进指南

## 一、 项目背景与核心诉求
在智能家居场景下，传统的云端 AI 面临着**网络延迟、隐私泄露风险、以及断网不可用**的致命痛点。
本项目旨在打造一个**纯本地运行、极低延迟、强隐私保护**的智能家居 AI 中枢（On-Device AI Agent），实现从自然语言指令到 IoT 硬件控制的端到端闭环。

## 二、 核心架构设计与技术决策

本项目的核心架构遵循了**分层解耦、端侧优先、规则兜底**的设计理念。
整个 Agent 被封装在 `on_device_agent` package 中，与业务 UI 实现了完美隔离。

### 1. 推理引擎层 (Inference Engine)
*   **技术选型**：摒弃了臃肿的 Python 栈，直接采用 `llama.cpp` 的 C++ 源码结合 Flutter FFI (Foreign Function Interface) 进行深度绑定。
*   **核心决策**：
    *   **GGUF 量化格式**：使用 Q4 甚至更低位数的量化模型（如 Gemma-2B/Qwen1.5-1.8B），将原本几十GB的模型压缩至 1~2GB，使其能够勉强塞进手机的 RAM 中。
    *   **Isolate 异步隔离**：在 `LlamaCppEngine` 中，将 C++ 模型的加载和推理全部放入 Dart 的后台 Isolate 中执行，绝对不阻塞 Flutter 的 Main UI Thread，保证了界面的 60fps 流畅度。

### 2. 确定性输出与防幻觉机制 (Determinism & Zero-Hallucination)
大模型直接控制硬件最可怕的是“幻觉”（即凭空编造不存在的设备）。
*   **技术决策**：**动态 GBNF (GGML BNF) 语法树约束**。
*   **实现细节**：
    *   在每次推理前，`SmartHomeAgent` 会向全局状态管理 (`DeviceManager`) 获取当前真实存在的设备列表。
    *   动态拼接出类似 `device_id ::= "\"light_1\"" | "\"ac_1\""` 的严格语法约束树，注入到 `llama.cpp` 的底层采样器中。
    *   从 Logits 层面上彻底掐断大模型输出非法字符的可能性，实现了 **100% 的 JSON 结构和设备实体解析准确率**。

### 3. 端侧轻量级 RAG (检索增强生成)
为了让 AI 拥有“记忆”，能够回答类似“昨天空调开了多久”的问题。
*   **技术决策**：使用高性能本地对象数据库 **Isar** 替代笨重的向量数据库。
*   **实现细节**：
    *   **无感落库**：每次执行成功的硬件指令，都会作为 `BehaviorLog` 存入 Isar。
    *   **意图拦截**：在 `handleUserQuery` 时，通过轻量级正则/关键词前置拦截查询意图。
    *   **上下文注入**：从 Isar 中查出最近的日志，转化为文本（如 `时间:xx 设备:xx 动作:xx`），作为 Context 塞入 System Prompt。
    *   **伪设备隔离**：在 GBNF 中加入 `"system"` 伪设备 ID，规定 AI 若为纯文本回复，则输出 `{"device_id": "system", "action": "reply", "value": "..."}`。在 `ActionExecutor` 解析时直接拦截放行，绝不触发真实的 IoT 控制。

### 4. UI 与体验层 (UX & Feedback)
*   **状态变化感知**：重构了 `ExecutionResult`，AI 助手在执行完毕后，UI 气泡内会渲染直观的**“操作前 vs 操作后”**的状态对比卡片。
*   **思维链透明化**：在聊天界面中加入了可折叠的“思维链 (Processing Steps)”面板，向用户透明展示 AI “理解意图 -> 组装上下文 -> 检索日志 -> 执行动作”的思考过程，极大提升了用户的信任感。
*   **性能监控面板**：引入 `PerformanceMetrics`，在 Debug 模式下 (`kDebugMode`)，每条回复下方会展示 TTFT (首字延迟)、总耗时和吞吐量 (Tokens/s)，为架构师和开发者提供了极佳的性能调优抓手。

## 三、 当前架构的局限性与后期迭代方向 (Next Steps)

虽然当前架构已经跑通了端侧大模型的闭环，但要走向真正的 Production-grade (生产级高可用)，团队在接下来的迭代中需要重点攻克以下难关：

### 1. 模型分发与生命周期管理 (Model OTA)
*   **现状**：目前假设模型直接打包在 assets 或本地路径中，导致 App 包体积极大。
*   **演进方向**：
    *   实现 **按需动态下发**。App 初始不含模型，用户开启 AI 功能时，使用断点续传下载 `.gguf` 到沙盒目录。
    *   引入 **硬件自适应探针**。启动时检测 RAM 大小和 NPU 算力：
        *   高端机 (>8GB RAM)：下载 3B 模型。
        *   中低端机 (<4GB RAM)：降级为 0.5B 模型或回退到纯规则引擎/云端 API。
    *   实现 **KV Cache 缓存复用**和**后台自动卸载模型**机制，防止触发系统 OOM-killer 导致应用闪退。

### 2. 异构计算与推理加速 (Hardware Acceleration)
*   **现状**：目前的 FFI 封装仅使用了 CPU 推理，Tokens/s 指标可能只有个位数，首字延迟过高。
*   **演进方向**：
    *   **iOS 端**：必须在 CMake 编译参数中强制开启 `GGML_METAL=1`，调用苹果的 GPU/Neural Engine。
    *   **Android 端**：集成 `GGML_VULKAN` 或高通/联发科的专有 NPU SDK (如 QNN/ExecuTorch)，这将使端侧推理速度提升 5~10 倍。

### 3. 端云协同网关架构 (Hybrid Routing)
*   **现状**：目前端侧模型要么处理，要么失败。无法回答通用百科或复杂计算。
*   **演进方向**：引入多层级路由意图网关 (Intent Router)。
    *   **Layer 1 (0ms)**：纯正则和关键词规则兜底（处理极简指令，如“关灯”）。
    *   **Layer 2 (~1s)**：端侧大模型（处理模糊意图、本地 RAG 查询、家居联动）。
    *   **Layer 3 (云端大模型)**：当端侧判断意图为 `out_of_domain`（非家居指令，如“今天天气如何”），透传给云端的大参数模型（如 GPT-4o），确保体验不中断。

### 4. 多模态交互与系统级融合 (Multi-modal)
*   **演进方向**：
    *   接入本地的 Whisper 模型实现真正的端侧离线语音唤醒和 ASR 识别。
    *   深度整合 iOS 的 App Intents (Siri Shortcuts) 或 Android 的 Assistant 机制，实现系统级唤醒。
    *   针对高危指令（如开大门、看监控），在 `ActionExecutor` 中挂起执行流，拉起系统的 FaceID/指纹进行本地生物认证。

## 结语
这个项目证明了**“大模型不再是云端的特权”**。通过严谨的架构设计（动态 GBNF + 轻量 RAG + Isolate 隔离），我们在端侧算力极度受限的环境下，成功构建了一个低延迟、零幻觉、强隐私的智能管家。
希望这份复盘能为团队在端侧 AI 领域的探索提供清晰的路线图！
